Binary Tree · Traversal

Postorder Traversal

Visit nodes in Left → Right → Root order. The root is always the last node visited.

Recursion DFS Interview Essential

Key concepts at a glance — for those who already know the basics.

01 · The rule

Left → Right → Root

Both subtrees are fully processed before you ever visit the current node. The root is always last.

02 · How to think

1
Recurse all the way into the left subtree
Do not visit the current node yet — go left first.
2
Then recurse into the right subtree
Still don't visit the current node — right subtree must finish too.
3
Only then visit (record) the current node
A parent is always visited after both its children.

03 · Code

def postorder(node, result):
    if node is None: return # base case
    postorder(node.left, result)  # 1. left
    postorder(node.right, result)# 2. right
    result.append(node.val)     # 3. root

04 · Complexity

Time
O(n)
Every node visited once.
Space
O(h)
Stack depth = tree height.

05 · Interview uses

Delete a tree — delete children before the parent (safe cleanup).

Evaluate expression trees — evaluate operands before the operator.

Height / depth of tree — compute children heights before parent.

Path sum problems — propagate info upward from leaves to root.

01 · Start here — what even is a binary tree?

Build the foundation first

Before traversal makes sense, you need to picture the structure. A binary tree is a collection of nodes where each node has at most two children — called the left child and the right child. The topmost node is called the root. Nodes with no children are called leaves.

The blue nodes are internal nodes (they have children). The amber nodes are leaves — they have no children. In postorder, leaves are always visited before their parents.

02 · What does "traversal" mean?

Visiting every node — in what order?

Traversal simply means visiting every node in the tree exactly once. The question is — in what order do you visit them? There are three classic DFS orders:

Preorder

Root→L→R

Inorder

L→Root→R

Postorder ← you are here

L→R→Root

Postorder means: for any node, always finish the entire left subtree first, then finish the entire right subtree, and only then visit the node itself.

03 · The real-world analogy

Think of it like cleaning up after a meal

Imagine you're cleaning up after a big family dinner. You can't wash and put away the serving bowls until all the smaller plates and glasses have been cleared first. The smallest items (leaves) get handled first, then the next level, and finally the big centrepiece (root) is dealt with last. Postorder traversal works the same way — children before parents, always.

The core insight: Postorder is the natural order for "bottom-up" computation — when a parent node needs information from both its children before it can do its own work. Deleting a tree, computing heights, and evaluating expression trees all follow this pattern.

04 · Understanding recursion here

What is the function actually doing?

Recursion is when a function calls itself. In postorder, the append (visit) is the very last line — it only runs after both recursive calls have returned. Here is how to read the code line by line:

def postorder(node, result):
    if node is None: return
    postorder(node.left, result)
    postorder(node.right, result)
    result.append(node.val)
Why do we check if node is None: return?
This is the base case — it stops the recursion. Without it, the function would keep calling itself forever. When a node has no left child, node.left is None, so we call postorder(None) which immediately returns. This is how recursion knows when to stop going deeper.
Why does the append line come last — not first or in the middle?
Because postorder means Left first, then Right, then Root. The append must wait for both subtree calls to complete. If you moved the append to the top you'd get preorder. If you moved it between the two calls you'd get inorder. Position of append defines the traversal type.
What does the recursion stack look like at the deepest point?
When we first reach node 4 (the leftmost leaf), the call stack holds one frame per level traversed down:
→ postorder(node=4)  ← currently here
   postorder(node=2)  ← waiting for left+right
   postorder(node=1)  ← original call

05 · Full trace — every step explained

Walking through the example tree

Let's trace every single action the algorithm takes on this tree, narrated step by step. The tree has root 1, left child 2 (with leaves 4 and 5), and right child 3 (with leaves 6 and 7).

Going

Enter postorder(1) — the root

We start at 1. Rule says go left first, so we call postorder(1.left) = postorder(2). Node 1 is now waiting for both subtrees to finish.

Going

Enter postorder(2)

At node 2. Go left first → call postorder(2.left) = postorder(4). Node 2 is now waiting.

Going

Enter postorder(4) — leftmost leaf

At node 4. Go left → postorder(None) → base case, returns immediately. Go right → postorder(None) → base case, returns immediately. Now both subtrees are done.

Visit

Record 4 → result = [4]

No left, no right — both done. Append 4. Return back to node 2. Now node 2 goes right.

Going

Enter postorder(5) — right child of 2

At node 5. Go left → None. Go right → None. Both subtrees done immediately.

Visit

Record 5 → result = [4, 5]

Node 5 is a leaf. Append 5. Return to node 2. Both children of 2 are now done.

Visit

Record 2 → result = [4, 5, 2]

Both subtrees of 2 are fully finished. Now visit node 2 itself. Return to node 1, which now goes right.

Going

Enter postorder(3) — right child of root

At node 3. Go left first → call postorder(6). Node 3 is now waiting.

Going

Enter postorder(6) — left leaf of 3

At node 6. Go left → None. Go right → None. Both done immediately.

Visit

Record 6 → result = [4, 5, 2, 6]

Node 6 is a leaf. Append 6. Return to node 3. Now node 3 goes right.

Going

Enter postorder(7) — right leaf of 3

At node 7. Go left → None. Go right → None. Both done immediately.

Visit

Record 7 → result = [4, 5, 2, 6, 7]

Node 7 is a leaf. Append 7. Return to node 3. Both children of 3 are now done.

Visit

Record 3 → result = [4, 5, 2, 6, 7, 3]

Both subtrees of 3 are fully finished. Visit node 3. Return to root (1). Both subtrees of 1 are now done.

Visit

Record 1 → result = [4, 5, 2, 6, 7, 3, 1] ✓

The root is the very last node visited. This is the defining property of postorder — the root always comes last.

06 · Common beginner mistakes

What trips people up

✕

Forgetting the base case

Without if node is None: return, the function crashes with a recursion error when it tries to access None.left.

✕

Mixing up the order

Moving the append before both calls gives preorder. Moving it between the calls gives inorder. In postorder, append must always be the very last line.

✕

Assuming root is first

Unlike preorder where the root appears first in the output, in postorder the root always appears last. This surprises many beginners.

✕

Not passing result by reference

In Python, lists are passed by reference — all recursive calls share the same list. If you create a new list inside each call instead, results will be lost.

07 · Complexity — the full explanation

Why O(n) time and O(h) space?

Time
O(n)
We visit every node exactly once — no node is skipped or revisited — so time grows linearly with the number of nodes.
Space
O(h)
The call stack holds one frame per level of depth. h = log n for balanced trees, h = n for skewed (like a linked list).

Interview tip: If they ask "what is the space complexity?", always clarify — O(log n) for a balanced tree, O(n) worst case for a skewed tree. Showing you know both impresses interviewers.

08 · Interview context

Where you'll actually use this

Delete / free a tree

You must delete children before the parent. Postorder ensures no parent is removed while its children still exist — safe and correct teardown.

Evaluate expression trees

Leaves are operands (numbers), internal nodes are operators (+, ×). Evaluate left and right subtrees first, then apply the operator at the root.

Compute height / diameter of a tree

To know a node's height, you need both children's heights first. Postorder gives you exactly that — children before parent, bottom to top.

Path sum / max path sum

Aggregate values from subtrees and pass them upward. This bottom-up pattern is the hallmark of postorder problems.
