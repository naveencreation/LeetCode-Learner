Binary Tree · Traversal

Preorder Traversal

Visit nodes in Root → Left → Right order. The root is always processed first, before any subtree.

Recursion DFS Interview Essential

Key concepts at a glance — for those who already know the basics.

01 · The rule

Root → Left → Right

The current node is always visited before exploring any of its children.

02 · How to think

1
Visit (record) the current node immediately
Before going anywhere — this is what makes it "pre" order.
2
Recurse into the left subtree
Handle the entire left branch, root-first, the same way.
3
Recurse into the right subtree
Apply Root → Left → Right recursively to the right branch.

03 · Code

def preorder(node, result):
    if node is None: return # base case
    result.append(node.val)     # 1. root ← first!
    preorder(node.left, result)  # 2. left
    preorder(node.right, result)# 3. right

04 · Complexity

Time
O(n)
Every node visited once.
Space
O(h)
Stack depth = tree height.

05 · Interview uses

Serialize a binary tree — preorder output + nulls fully reconstructs the tree.

Copy / clone a tree — root-first creation means children can be attached right after.

Print directory structure — preorder mirrors how file systems display nested folders.

Find a path to a node — root-to-leaf path tracking is naturally preorder.

01 · Start here — what even is a binary tree?

Build the foundation first

Before traversal makes sense, you need to picture the structure. A binary tree is a collection of nodes where each node has at most two children — called the left child and the right child. The topmost node is called the root. Nodes with no children are called leaves.

The purple node is the root. The green nodes are leaves — they have no children. Every other node is an internal node.

02 · What does "traversal" mean?

Visiting every node — in what order?

Traversal simply means visiting every node in the tree exactly once. The question is — in what order do you visit them? There are three classic DFS orders:

Preorder ← you are here

Root→L→R

Inorder

L→Root→R

Postorder

L→R→Root

Preorder means: for any node, visit it immediately, then explore the entire left subtree, then the entire right subtree. The "pre" in preorder means the root comes before its children.

03 · The real-world analogy

Think of it like exploring a cave system

Imagine you're a cave explorer. Every time you enter a new chamber, you mark it on your map immediately — before heading deeper. Then you go left as far as you can, marking each chamber as you enter. Only after you've fully explored the left tunnel do you backtrack and explore the right. Preorder traversal works exactly the same way: mark now, explore later.

The key insight: Preorder naturally produces a "top-down" view of the tree. The root always appears first in the output, making preorder ideal for serializing and reconstructing trees.

04 · Understanding recursion here

What is the function actually doing?

Recursion is when a function calls itself. In preorder, the crucial difference from inorder is that append happens first — before any recursive calls. Here is how to read the code line by line:

def preorder(node, result):
    if node is None: return
    result.append(node.val)
    preorder(node.left, result)
    preorder(node.right, result)
Why does append come before the recursive calls?
Because "pre" means before. We record the current node's value before we dive into either subtree. This is the only difference from inorder — in inorder, the append is sandwiched between the two calls. Flip it to first and you get preorder.
Why do we still check if node is None: return?
This is the base case — it stops the recursion. Without it, when we reach a leaf and try to call preorder(leaf.left), we'd be calling preorder(None), which would then try to access None.val and crash. The base case makes every leaf safe.
What does the recursion stack look like when we first reach node 4?
In preorder we've already recorded nodes 1, 2, and 4 before the stack gets this deep. When we call preorder(4.left), the stack is:
→ preorder(node=None)  ← base case, returns
   preorder(node=4)   ← already recorded 4
   preorder(node=2)   ← already recorded 2
   preorder(node=1)   ← already recorded 1

05 · Full trace — every step explained

Walking through the example tree

Let's trace every single action the algorithm takes on this tree, narrated step by step. Expected output: [1, 2, 4, 5, 3, 6, 7]

Visit

Record 1 → result = [1]

Enter preorder(1) — the root. We record it immediately before doing anything else. Then recurse left → preorder(2).

Visit

Record 2 → result = [1, 2]

Enter preorder(2). Record it immediately. Then recurse left → preorder(4). Node 2's right call is queued for later.

Visit

Record 4 → result = [1, 2, 4]

Enter preorder(4) — a leaf. Record it. preorder(None) for left returns instantly. preorder(None) for right returns instantly. Done with node 4, return to node 2.

Going

Back at node 2 — now go right

Left subtree of 2 is fully done. Now execute the right call → preorder(5).

Visit

Record 5 → result = [1, 2, 4, 5]

Enter preorder(5) — a leaf. Record it. Both children are None, so it returns. Entire left subtree of root (node 1) is now done.

Going

Back at root node 1 — now go right

Left subtree of 1 is fully done. Now execute the right call → preorder(3).

Visit

Record 3 → result = [1, 2, 4, 5, 3]

Enter preorder(3). Record it immediately. Then recurse left → preorder(6).

Visit

Record 6 → result = [1, 2, 4, 5, 3, 6]

Enter preorder(6) — a leaf. Record it. Both children are None. Return to node 3.

Visit

Record 7 → result = [1, 2, 4, 5, 3, 6, 7] ✓

Back at node 3, now go right → preorder(7). Node 7 is a leaf. Record it. All done — this is the final preorder sequence.

06 · Common beginner mistakes

What trips people up

✕

Forgetting the base case

Without if node is None: return, the function tries to access None.val and crashes. Unlike inorder, in preorder this crash happens before the append — so you'd never even see partial output.

✕

Moving the append to the middle or end

Moving append between the two calls gives inorder. Moving it after both calls gives postorder. The position of append defines which traversal you're doing.

✕

Thinking preorder = BFS (level-order)

Preorder is a DFS — it dives deep left before going right. BFS visits nodes level by level using a queue. They look similar on paper but produce very different outputs.

✕

Confusing serialization with sorted output

Preorder does not give sorted output even for a BST. That's inorder's job. Preorder's superpower is tree reconstruction — not sorting.

07 · Complexity — the full explanation

Why O(n) time and O(h) space?

Time
O(n)
We visit every node exactly once — no node is skipped or revisited — so time grows linearly with the number of nodes.
Space
O(h)
The call stack holds one frame per level of depth. h = log n for balanced trees, h = n for skewed (like a linked list).

Interview tip: If they ask "what is the space complexity?", always clarify — O(log n) for a balanced tree, O(n) worst case for a skewed tree. Showing you know both impresses interviewers.

08 · Interview context

Where you'll actually use this

Serialize and deserialize a binary tree

Preorder output (with null markers) uniquely encodes a tree. Feed it back in to rebuild the exact same structure.

Clone / deep copy a tree

Create the root node first, then recursively create left and right children — exactly the preorder pattern.

Find root-to-leaf paths

Track the current path as you go. Each time you reach a leaf in preorder, you've collected a complete root-to-leaf path.

Construct tree from preorder + inorder

The first element of preorder is always the root. Use inorder to find its left/right split. Repeat recursively — classic divide and conquer interview problem.
