Construct Binary Tree from Inorder & Postorder Traversal

Given the inorder and postorder traversal arrays of a binary tree, reconstruct and return the original tree.

Divide & Conquer Hash Map Recursive DFS Medium

Key concepts at a glance — for those who already know the basics.

01 · The two key facts

Postorder's last element is always the root.
Inorder splits the tree: everything left of root = left subtree, everything right = right subtree.

These two facts together let you recursively rebuild the entire tree in one DFS pass.

02 · How to think (4 steps, repeated recursively)

1
Pop the last element of postorder — that is the current root
Postorder = Left, Right, Root — so the root is always at the end.
2
Find the root's index in inorder (use a hash map for O(1))
Everything to its left in inorder belongs to the left subtree; everything to its right belongs to the right subtree.
3
Calculate the size of the left subtree from the inorder split
left_size = inorder_root_idx - in_start. Use this to split the postorder array too.
4
Recurse: build right subtree first, then left subtree
Build right before left because we're consuming postorder from the back — right subtree's root comes before left subtree's root as we move leftward.

03 · Code (Python)

class Solution:
def buildTree(self, inorder, postorder):
idx_map = {val: i for i, val in enumerate(inorder)}
 
def build(in_l, in_r, post_l, post_r):
if in_l > in_r: return None # base case
 
root_val = postorder[post_r] # last of postorder = root
root = TreeNode(root_val)
mid = idx_map[root_val] # root's position in inorder
lsize = mid - in_l # size of left subtree
 
root.left = build(in_l, mid-1, post_l, post_l+lsize-1)
root.right = build(mid+1, in_r, post_l+lsize, post_r-1)
return root
 
return build(0, len(inorder)-1, 0, len(postorder)-1)

The key insight: Postorder tells us who the root is (always the last element). Inorder tells us what belongs to each subtree (everything left of root = left subtree, everything right = right subtree). Together they uniquely determine the tree.

02 · The core mechanism — how one root splits both arrays

Why build right before left? We consume postorder from right to left. After consuming the overall root at post_r, the right subtree's root sits just to its left at post_r - 1. So we must build the right subtree first, then the left.

03 · Why we precompute a hash map

O(n²) vs O(n) — one data structure changes everything

At every recursive call we need to find the root's index in inorder. Without a hash map, we do a linear scan — O(n) per call, O(n) calls → O(n²) total. With a hash map built once at the start, every lookup is O(1) — making the whole algorithm O(n).

Without hash map

Linear search through inorder each time. n recursive calls × O(n) search = O(n²) time. Acceptable for n ≤ 100 but fails at scale.

With hash map ✓

Precompute val → index once in O(n). Every lookup is O(1). n calls × O(1) = O(n) total. Always prefer this in interviews.

# Build once — O(n)
idx_map = {val: i for i, val in enumerate(inorder)}
# e.g. {9:0, 3:1, 15:2, 20:3, 7:4}
 
# Use inside recursion — O(1) each time
mid = idx_map[root_val]

04 · Full code — every line explained

Reading the solution line by line

class Solution:
def buildTree(self, inorder, postorder):
# ① precompute inorder index map
idx_map = {val: i for i, val in enumerate(inorder)}
 
def build(in_l, in_r, post_l, post_r):
# ② base case: empty slice
if in_l > in_r: return None
 
# ③ root = last element of current postorder slice
root_val = postorder[post_r]
root = TreeNode(root_val)
 
# ④ find root in inorder → splits left / right
mid = idx_map[root_val]
lsize = mid - in_l
 
# ⑤ recurse: left subtree
root.left = build(in_l, mid-1, post_l, post_l+lsize-1)
# ⑥ recurse: right subtree
root.right = build(mid+1, in_r, post_l+lsize, post_r-1)
return root
 
return build(0, len(inorder)-1, 0, len(postorder)-1)
① Why build idx_map before the recursive function?
We only need to build it once — it never changes across recursive calls. Building it inside the recursive function would re-create a new dict every call, adding O(n) work per call and ballooning total time to O(n²). One dictionary, built once outside, is shared by all recursive calls via Python's closure.
② What exactly does in_l > in_r mean and when does it trigger?
It means the inorder slice we were handed is empty — there are no nodes left to build in this part of the tree. This happens when a node is a leaf (both its left and right children are null). We return None, which the parent will assign to root.left or root.right.
⑤⑥ How are the postorder slice boundaries for left and right computed?
Left subtree has lsize nodes. In postorder (L R Root), the left subtree occupies the first lsize positions of the current slice: [post_l .. post_l+lsize-1]. The right subtree fills the rest, just before the root: [post_l+lsize .. post_r-1]. The root itself was at post_r and is now consumed.

05 · Real-world analogy

Think of it like reading a nested table of contents

Imagine a book's table of contents (inorder) and a list of chapter summaries written with the deepest sub-chapters first and the top chapter last (postorder). The last summary tells you the name of the current chapter — that's your root. The table of contents then tells you which sub-chapters belong to the left half and which to the right half of that chapter. You repeat this for each half, always looking at the deepest summary available and using the table of contents to split. Eventually every chapter is placed perfectly in the hierarchy.

06 · Full recursive trace

Walking through inorder=[9,3,15,20,7], postorder=[9,15,7,20,3]

We'll trace every call and show the array slices and tree nodes being built. Index notation: in[l..r], post[l..r].

Call 1

build(in[0..4], post[0..4])

postorder[4] = 3 → create root node 3. Find 3 in inorder: index 1. lsize = 1−0 = 1.
Left subtree uses in[0..0], post[0..0]. Right subtree uses in[2..4], post[1..3].

Call 2 (Left)

build(in[0..0], post[0..0]) — building node 3's left child

postorder[0] = 9 → create node 9. Find 9 in inorder: index 0. lsize = 0−0 = 0.
Left: build(in[0..-1]) → base case → None. Right: build(in[1..0]) → base case → None.
Node 9 is a leaf. Return node 9.

Call 3 (Right)

build(in[2..4], post[1..3]) — building node 3's right child

postorder[3] = 20 → create node 20. Find 20 in inorder: index 3. lsize = 3−2 = 1.
Left subtree uses in[2..2], post[1..1]. Right subtree uses in[4..4], post[2..2].

Call 4 (Left of 20)

build(in[2..2], post[1..1]) — building node 20's left child

postorder[1] = 15 → create node 15. Find 15 in inorder: index 2. lsize = 0.
Both children are base cases → None. Node 15 is a leaf. Return node 15.

Call 5 (Right of 20)

build(in[4..4], post[2..2]) — building node 20's right child

postorder[2] = 7 → create node 7. Find 7 in inorder: index 4. lsize = 0.
Both children → None. Node 7 is a leaf. Return node 7.

Result ✓

Tree fully built: 3 → {left: 9, right: 20 → {left: 15, right: 7}}

All 5 nodes created, 5 recursive calls made (plus leaf base-case calls that immediately return None). The final tree matches the expected output exactly.

08 · Common mistakes

Errors to avoid in interviews

✕

Building left before right

In postorder, right subtree nodes always come immediately before the root (after all left subtree nodes). If you build left first and consume postorder from the back, you'll consume right subtree elements for the left subtree — producing a completely wrong tree. Always build right first, then left when consuming postorder from the right.

✕

Wrong postorder boundary for the right subtree

The right subtree's postorder slice ends at post_r - 1 (just before the current root). A common off-by-one is using post_r instead — this includes the current root in the child's slice, causing it to create a duplicate node.

✕

Skipping the hash map and using linear search

Calling inorder.index(root_val) inside the recursion is O(n) per call → O(n²) overall. For n=3000 (the constraint limit) that's 9 million operations unnecessarily. Always precompute the index map.

✕

Confusing this with LC 105 (Preorder + Inorder)

In LC 105 the root is the first element of preorder; here the root is the last element of postorder. The inorder splitting logic is identical, but you consume preorder from the front and build left-before-right, while here you consume postorder from the back and build right-before-left.

09 · Complexity — the full explanation

Why O(n) time and O(n) space?

Time
O(n)
Each of the n nodes triggers exactly one recursive call. Each call does O(1) work (one hash map lookup, one TreeNode creation). Total: O(n).
Space
O(n)
O(n) for the idx_map. O(h) for the recursion stack, where h = height (O(log n) balanced, O(n) skewed). Total dominated by the map: O(n).

Interview tip: If asked to reduce space, you can eliminate the hash map by passing inorder as a slice and using list.index(). That drops space to O(h) for the stack but raises time to O(n²). Explain the trade-off clearly — interviewers reward this kind of reasoning.

10 · Interview Q&A

Questions you'll actually get asked

Can you solve this without a hash map?
Yes — replace idx_map[root_val] with inorder.index(root_val) inside the recursive call. This is simpler to write but O(n) per call, making total time O(n²). For the given constraint (n ≤ 3000) it still passes, but the hash map version is the expected "optimal" answer in an interview.
Why can't you reconstruct a unique tree from postorder alone?
Postorder tells you the root and the relative order of all other nodes, but not the boundary between the left and right subtrees. Without inorder, you cannot determine how many of the remaining nodes belong to the left child vs the right child. For example, postorder = [1, 2] could be either a tree with root 2 and left child 1, or root 2 and right child 1.
What changes if the tree has duplicate values?
The problem guarantees unique values, which is why the hash map works — each value maps to exactly one inorder index. With duplicates, the inorder lookup becomes ambiguous: idx_map[root_val] would return only one of the duplicate positions, potentially giving the wrong split. Solving the problem with duplicates requires additional structural information or a different approach entirely.
How does this differ from LC 105 (Preorder + Inorder)?
The splitting logic is identical — inorder is used the same way in both. The only difference is where the root comes from (first of preorder vs last of postorder) and the recursive order (left-first for preorder, right-first for postorder). If you internalize the inorder-splitting idea, both problems reduce to the same pattern with a minor variation in how you index into the second array.
