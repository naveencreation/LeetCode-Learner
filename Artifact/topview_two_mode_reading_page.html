Top View of Binary Tree

The first node visible at each horizontal distance when the tree is seen from the top.

BFS + HashMap Horizontal Distance Interview Essential

Key concepts at a glance — for those who already know the basics.

01 · The rule

First node seen at each horizontal distance from the root

Assign root HD=0. Left child gets HD−1, right child HD+1. For each HD, only the shallowest (topmost) node is visible — if an HD column already has an entry, ignore any deeper node in that column.

02 · How to think

1
Do a BFS, tracking each node's horizontal distance (HD)
Root starts at HD = 0. Left child: HD − 1. Right child: HD + 1.
2
For every HD, record only the first node you encounter — never overwrite
BFS visits nodes top-to-bottom, so the first node at any HD is the shallowest. That's the top view.
3
Read the map sorted by HD from min to max
Gives you the top view left-to-right as it would appear looking down at the tree from above.

03 · Code

from collections import deque
 
def top_view(root):
    if not root: return []
    hd_map = {}                        # hd → node value
    queue = deque([(root, 0)])        # (node, hd)
    while queue:
        node, hd = queue.popleft()
        if hd not in hd_map:       # only record the first (topmost)
            hd_map[hd] = node.val
        if node.left: queue.append((node.left, hd - 1))
        if node.right: queue.append((node.right, hd + 1))
    return [hd_map[k] for k in sorted(hd_map)]

01 · The core idea

What does "top view" actually mean?

Imagine hovering directly above a binary tree and looking straight down. For every vertical column, only the highest node is visible to you — every node below it is completely obscured. That's the top view: for each horizontal column, report the first (shallowest) node you encounter.

The key concept is Horizontal Distance (HD). Root is at HD = 0. Going left subtracts 1; going right adds 1. Two nodes that share the same HD fall in the same vertical column — only the one at the smallest depth (encountered first in BFS) is visible from the top.

02 · Why BFS, not DFS?

Two approaches — one is much cleaner

BFS ← natural fit

Processes nodes level by level (top to bottom). The first time a new HD is encountered, it is guaranteed to be the shallowest — no depth check needed. Just write once and never overwrite.

DFS (recursive)

Must carry depth explicitly and only record when the new depth is strictly less than the stored depth. More error-prone and harder to reason about for tied depths.

Interview tip: Always lead with BFS. If pressed for a DFS alternative, say: "I'd track (hd, depth) in the map and only write when the new node's depth is less than the stored depth — never for equal depth, because equal depth means a deeper or same-level duplicate, not a higher one."

03 · The real-world analogy

Think of it like an aerial photograph

Hang the tree from the ceiling with the root at the top. Shine a light upward from below. Each node blocks every node directly above it in the same column. The top view is the set of nodes that are not blocked by anything higher up — the ones you'd see in the aerial photo. One node per vertical column, always the highest one.

Key insight: The critical difference from the bottom view is a single if hd not in hd_map guard. Without it, you'd always be overwriting with deeper nodes — that gives you the bottom view. The guard locks in the first write, giving you the top view.

04 · Understanding the code line by line

What the code is actually doing

The key change from bottom view is the conditional write: if hd not in hd_map. Because BFS is level-by-level, the very first time we see a new HD column, the current node is the topmost one. We lock it in and refuse to overwrite — that's the entire algorithm.

from collections import deque
 
def top_view(root):
    if not root: return []
    hd_map = {}
    queue = deque([(root, 0)])
    while queue:
        node, hd = queue.popleft()
        if hd not in hd_map: # first node at this HD = topmost
            hd_map[hd] = node.val
        if node.left: queue.append((node.left, hd - 1))
        if node.right: queue.append((node.right, hd + 1))
    return [hd_map[k] for k in sorted(hd_map)]
Why do we check if hd not in hd_map instead of just writing every time?
Because BFS is level-by-level, the very first node we encounter at any HD is guaranteed to be the shallowest (topmost) in that column. If we kept overwriting, we'd end up with the deepest node (bottom view). The condition locks in the first — and only the first — entry per HD column.
What if two nodes share the same HD and the same depth?
For the top view, this doesn't change anything — the first node at that HD is already locked in, regardless of ties at lower levels. The check if hd not in hd_map fires exactly once per column, at the shallowest depth, and subsequent nodes (even at the same depth in a different branch) are simply skipped.
Why sort the map keys at the end?
Python dicts preserve insertion order, but HD keys are inserted in BFS order (not necessarily sorted left-to-right). To return the top view from left to right (min HD to max HD), we must sort the keys. This is the same as in bottom view and adds O(n log n) to time complexity.

05 · Full trace — every step explained

Walking through the example tree

Tree: root=1 (HD=0). Left=2 (HD=−1) with children 4 (HD=−2) and 5 (HD=0). Right=3 (HD=+1) with children 6 (HD=0) and 7 (HD=+2). Let's trace every BFS action.

Start

Queue = [(1, 0)]

Root enters the queue with HD = 0.

Write

Dequeue (1, HD=0) → hd_map = {0: 1}

HD=0 is new — record node 1. Enqueue children: (2, −1) and (3, +1).

Write

Dequeue (2, HD=−1) → hd_map = {0:1, −1:2}

HD=−1 is new — record node 2. Enqueue (4, −2) and (5, 0).

Write

Dequeue (3, HD=+1) → hd_map = {…, +1:3}

HD=+1 is new — record node 3. Enqueue (6, 0) and (7, +2).

Write

Dequeue (4, HD=−2) → hd_map = {…, −2:4}

HD=−2 is new — record node 4. Leaf — nothing enqueued.

Skip

Dequeue (5, HD=0) → hd_map unchanged

HD=0 already has node 1. Node 5 is deeper — it's invisible from the top. Leaf — nothing enqueued.

Skip

Dequeue (6, HD=0) → hd_map unchanged

HD=0 still has node 1. Node 6 is also depth 2 — also invisible from the top. Leaf — nothing enqueued.

Write

Dequeue (7, HD=+2) → hd_map = {−2:4, −1:2, 0:1, +1:3, +2:7}

Queue empty. Sort keys → Top View = [4, 2, 1, 3, 7] ✓

06 · Common beginner mistakes

What trips people up

✕

Overwriting unconditionally — writing the bottom view instead of top

The single most common mistake. Omitting the if hd not in hd_map guard means you always overwrite with a deeper node — that's bottom view behaviour. The guard is the entire difference between the two problems.

✕

Using DFS without a depth check

In DFS, traversal order is not depth-ordered. You must track depth explicitly and only record when new_depth < stored_depth. If you use DFS and just check if hd not in hd_map, you'll record whichever DFS path arrives first — not necessarily the shallowest node.

✕

Confusing top view with root-to-leaf path or level-order

Top view is per-column (per HD), not per-level. A skewed tree where every node veers right will have a top view that looks like just the leftmost spine — one node per HD column, not one per level.

✕

Forgetting to sort the map keys before returning

Insertion order in the map depends on BFS traversal, not HD order. Without sorted(hd_map), the result may be scrambled. Always sort by HD to produce a left-to-right answer.

07 · Complexity — the full explanation

Why O(n log n) time and O(n) space?

Time
O(n log n)
BFS is O(n) — each node enqueued and dequeued exactly once. The conditional write is O(1). Sorting HD keys at the end adds O(n log n). If unsorted output is acceptable, the algorithm is O(n).
Space
O(n)
The queue holds at most O(w) nodes (width of the widest level). The HashMap holds at most n entries. Both are bounded by O(n).

Interview tip: Mention both time components: "O(n) for the BFS plus O(n log n) for sorting HD keys — dominated by the sort. If the interviewer asks for O(n), suggest a TreeMap (Java) or SortedDict (Python's sortedcontainers) for sorted insertion, removing the final sort step entirely."

08 · Interview context

Where you'll actually use this

Bottom View of Binary Tree

Identical BFS + HD setup — but remove the if hd not in hd_map guard and always overwrite. Last node per HD (deepest) is the bottom view.

Vertical Order Traversal (LeetCode 987)

Same HD tracking but collect all nodes per column, then sort by (row, value). Top view is the vertical-order minimum row per column — a special case.

Left/Right View of Binary Tree

Related concept but uses depth (level) rather than HD. Left view = first node at each level; Right view = last node at each level. BFS pattern is the same.

Width of Binary Tree at each level

The HD range (max HD − min HD + 1) gives the width of a level. Reuse the same BFS + HD framework from the top view solution directly.
