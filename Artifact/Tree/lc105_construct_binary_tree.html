Construct Binary Tree from Preorder and Inorder Traversal

Given the preorder and inorder traversal arrays of a binary tree, reconstruct and return the original tree.

Recursive DFS Divide & Conquer HashMap O(1) Lookup Medium

Key concepts at a glance — for those who already know the basics.

01 · The two key facts

Preorder

First element is always the root

Inorder

Root splits left / right subtrees

Find the root in the inorder array → everything to its left belongs to the left subtree, everything to its right belongs to the right subtree. This split then applies recursively.

02 · How to think

1
The root is always preorder[0]
Preorder visits root first. So the very first element in the current preorder slice is the root of the current subtree.
2
Find that root value in the inorder array
Use a HashMap for O(1) lookup. Its index mid tells us how many nodes are in the left subtree: left_size = mid − inorder_start.
3
Slice both arrays and recurse
Left subtree uses the next left_size elements from preorder and everything left of mid in inorder. Right subtree gets the rest.
4
Base case: empty slice → return None
When the current subproblem has no elements, there's no node to create.

03 · Code (Python)

class Solution:
def buildTree(self, preorder, inorder):
idx_map = {val: i for i, val in enumerate(inorder)}
 
def build(pre_l, pre_r, in_l, in_r):
if pre_l > pre_r: return None # empty subproblem
 
root_val = preorder[pre_l]          # always the root
mid = idx_map[root_val]       # root's position in inorder
left_size = mid - in_l              # nodes in left subtree
 
root = TreeNode(root_val)
root.left = build(pre_l + 1, pre_l + left_size,     in_l, mid - 1)
root.right = build(pre_l + left_size + 1, pre_r, mid + 1, in_r)
return root
 
return build(0, len(preorder) - 1, 0, len(inorder) - 1)

06 · Complexity

Time
O(n)
Each of the n nodes is created exactly once. HashMap lookup for the inorder index is O(1), so total is O(n).
Space
O(n)
O(n) for the HashMap and O(h) for the recursion stack. Worst case (skewed tree) h = O(n), so overall O(n).

01 · What do preorder and inorder actually mean?

The traversal properties that make reconstruction possible

Before solving the problem, you need to internalise what each traversal order encodes. These aren't arbitrary sequences — each one encodes structural information that the other alone cannot provide.

Preorder (Root → Left → Right): The root of any subtree is always visited first. So preorder[0] is the root of the whole tree. After recursing into the left subtree (which comes next in preorder), the right subtree follows — in exactly the same pattern. This tells you which node is the root at each level.

Inorder (Left → Root → Right): The root is visited after the entire left subtree and before the entire right subtree. This means every value to the left of the root in the inorder array belongs to the left subtree, and every value to the right belongs to the right subtree. This tells you how many nodes are on each side.

Together these two facts completely determine the tree. Neither alone is sufficient: preorder alone doesn't tell you where the left subtree ends, and inorder alone doesn't tell you which node is the root.

02 · The split visualised — level by level

How each recursive call carves out its portion

The key to understanding this problem is seeing how the two array windows shrink with each recursive call. At every level you identify the root, measure the left subtree size, then pass the correct slices to each child.

Pattern: At every call, preorder[pre_l] is the root. Its position mid in inorder gives left_size = mid − in_l. The left child gets preorder slice [pre_l+1 .. pre_l+left_size]; the right child gets [pre_l+left_size+1 .. pre_r]. Inorder slices around mid the same way.

03 · Why the HashMap matters

The difference between O(n²) and O(n)

Every recursive call needs to find the root's position in the inorder array. Without a HashMap, you'd scan the inorder array each time — O(n) per call, O(n) calls → O(n²) total. With a HashMap built once upfront, each lookup is O(1).

Without HashMap — O(n²)

Linear scan of inorder at each level. For a skewed tree with n levels, that's 1+2+…+n = O(n²) total comparisons.

With HashMap ← O(n)

Build {value: index} once in O(n). Every subsequent lookup is O(1). Total: O(n) for the HashMap + O(n) recursive calls = O(n).

Interview tip: Always mention the HashMap optimisation. If you present the naive slicing version first, immediately follow with: "This is O(n²) due to repeated index scans. I can improve it to O(n) by pre-building a value-to-index map for the inorder array and passing index boundaries instead of slicing."

04 · Understanding the index arithmetic

How to compute the four boundaries correctly

This is where most people get confused. The four parameters — pre_l, pre_r, in_l, in_r — represent the current window of each array. At every call you carve two smaller windows from these and pass them to the left and right children.

# Given: pre_l, pre_r, in_l, in_r for the current call
root_val = preorder[pre_l]          # always the first in pre-window
mid = idx_map[root_val]       # root's position in in-window
left_size = mid - in_l              # how many nodes in left subtree
 
# Left child windows:
pre: [pre_l+1 .. pre_l+left_size]  # skip root, take left_size
in: [in_l .. mid-1]           # everything left of mid
 
# Right child windows:
pre: [pre_l+left_size+1 .. pre_r]  # skip root + left_size
in: [mid+1 .. in_r]              # everything right of mid
Why is the left preorder window [pre_l+1 .. pre_l+left_size] and not just until mid?
Because mid is an index in the inorder array, not the preorder array. The preorder and inorder windows are tracking different things. We know the left subtree has exactly left_size = mid − in_l nodes, so the left preorder window starts right after the root (pre_l+1) and spans left_size elements.
What guarantees the preorder window sizes match the inorder window sizes?
Both traversals visit exactly the same set of nodes — the problem guarantees this. So any subtree always has the same number of nodes in its preorder slice and its inorder slice. If the left subtree has 2 nodes in the inorder window, it also occupies exactly 2 slots in the preorder window (immediately after the root).
Why use index boundaries instead of actually slicing the arrays?
Array slicing in Python creates new lists — that's O(k) time and O(k) space per call, where k is the slice length. With index boundaries, each call is O(1) additional work (just arithmetic and one HashMap lookup), and the original arrays are never copied. This is what makes the full solution genuinely O(n) time and O(n) space (dominated by the HashMap and output tree, not recursive copying).

As each call returns its node, the parent attaches it. Node(20).left = Node(15), Node(20).right = Node(7). Node(3).left = Node(9), Node(3).right = Node(20). Final root is Node(3).

06 · Real-world analogy

Think of it like reconstructing a chapter from two indexes

Imagine a book's table of contents lists chapters in the order they appear (preorder), always starting with the current section heading. A separate alphabetical index lists every topic and tells you what's to the left and right of each main heading in reading order (inorder). To reconstruct the book's hierarchy: the first entry in the table of contents is always the top-level heading. Find it in the alphabetical index — everything before it belongs to the left branch, everything after belongs to the right branch. Repeat this for each sub-section.

07 · Common mistakes

Errors to avoid in interviews

✕

Slicing arrays instead of passing index boundaries

Using buildTree(preorder[1:mid+1], inorder[:mid]) creates new arrays at every call — O(n) per call, O(n²) total. Always work with pre_l, pre_r, in_l, in_r index pointers.

✕

Using the inorder mid directly as the preorder boundary

mid is a position in the inorder array. The preorder left boundary is pre_l + left_size, where left_size = mid − in_l. Confusing the two gives wrong boundaries and a corrupted tree.

✕

Forgetting the base case pre_l > pre_r

Without this, the recursion continues into invalid ranges, causing index errors or infinite loops. The base case is the most critical line in the function.

✕

Skipping the HashMap and scanning inorder linearly

Using inorder.index(root_val) inside the recursive function is O(n) per call. With n recursive calls, this is O(n²). Build the idx_map once before recursion begins.

08 · Complexity — the full explanation

Why O(n) time and O(n) space?

Time
O(n)
n recursive calls, one per node. Each call does O(1) work — one preorder access and one O(1) HashMap lookup. Total: O(n).
Space
O(n)
O(n) for the HashMap. O(h) for the recursion stack (h = height). O(n) for the output tree. Dominated by the HashMap and output: O(n) overall.

Interview tip: Break down space as: "O(n) for the HashMap built upfront, plus O(h) for the recursion stack (O(log n) balanced, O(n) skewed), plus O(n) for the constructed tree itself — dominated by O(n) overall." This level of precision impresses interviewers.

09 · Interview Q&A

Questions you'll actually get asked

Can you reconstruct a binary tree from just preorder, or just inorder alone?
No — you need both. Preorder alone tells you the root at each level but not how many nodes are in each subtree. Inorder alone tells you the relative left/right positions but not which node is the root. Together they uniquely determine the tree (given unique node values).
What if the tree has duplicate values?
This algorithm requires unique values. The HashMap maps each value to exactly one inorder index. With duplicates, the same value could appear at multiple inorder positions, making the root-finding ambiguous. LeetCode 105 guarantees unique values in its constraints.
What's the equivalent problem with postorder instead of preorder?
LeetCode 106 — Construct Binary Tree from Inorder and Postorder Traversal. The logic is symmetric: in postorder the root is always the last element (instead of first), so you read from the end of the postorder array and construct the right subtree before the left.
Is there an iterative solution?
Yes, using an explicit stack to simulate the recursion. You push nodes and track whether you've set their left child yet. It's significantly more complex to implement correctly than the recursive version and provides no asymptotic improvement. In interviews, the recursive solution with index boundaries is preferred.
