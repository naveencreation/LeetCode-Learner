Convert BST to Sorted Doubly Linked List

In-place convert a BST into a circular doubly linked list sorted in ascending order — reusing the existing left/right pointers as prev/next.

In-order DFS Two Pointers In-place Medium

Key concepts at a glance — for those who already know the basics.

01 · The rule

In-order traversal of a BST visits nodes in sorted order — reuse left/right as prev/next

Do an in-order DFS, tracking the previously visited node. At each node, set prev.right = curr and curr.left = prev. After the traversal, connect the head and tail to form the circular link.

02 · How to think

1
Run in-order DFS (left → node → right)
This visits nodes in ascending order, which is exactly the sorted order we need for the DLL.
2
Maintain two pointers: head and prev
head = the very first node visited (smallest). prev = the last node linked so far.
3
Link prev → curr and curr → prev in-place
Set prev.right = curr (next pointer) and curr.left = prev (prev pointer). Reuse existing pointers — no new nodes.
4
After traversal, connect head ↔ tail to close the circle
Set head.left = prev (tail) and prev.right = head. Return head.

03 · Code (Python)

def treeToDoublyList(root):
if not root: return None
 
head = prev = None # head = smallest, prev = last linked
 
def inorder(node):
nonlocal head, prev
if not node: return
 
inorder(node.left) # 1. recurse left subtree
 
if prev: # 2. link prev ↔ curr
prev.right = node
node.left = prev
else: # first node visited = head
head = node
prev = node # advance prev
 
inorder(node.right) # 3. recurse right subtree
 
inorder(root)
 
# 4. close the circle: head ↔ tail
head.left = prev
prev.right = head
return head

01 · What is the problem really asking?

Three requirements packed into one sentence

The problem asks you to convert a BST into a circular doubly linked list, in-place, with nodes in ascending sorted order. Each requirement has implications:

In-place: you cannot allocate a new list or new nodes. You must reuse the existing tree nodes — the left pointer becomes the "prev" pointer and the right pointer becomes the "next" pointer.

Sorted: the DLL must be in ascending order. A BST's in-order traversal (left → node → right) visits nodes in sorted ascending order — that's the key insight.

Circular: the tail's next must point back to the head, and the head's prev must point to the tail. This is done in a single O(1) step after the full traversal.

The entire algorithm reduces to: run in-order DFS, stitch nodes together as you visit them, then close the circle at the end.

02 · Why in-order DFS?

The traversal order that gives you sorted output for free

In a BST, for every node: all values in its left subtree are smaller, and all values in its right subtree are larger. In-order DFS (recurse left → visit current → recurse right) naturally visits nodes from smallest to largest.

Key insight: The root (4) is visited third — not first. In-order traversal goes all the way down the left before visiting the current node. This is why pre-order or post-order won't work here — only in-order produces the globally sorted sequence.

03 · How pointers are reused in-place

Left = prev, Right = next — during and after conversion

Each tree node has a left and right pointer. The problem tells us to reuse them: left becomes the "previous" pointer in the DLL, and right becomes the "next" pointer. No new memory is allocated.

During the traversal, we only write to node.left and node.right — we never read the original child pointers at the "linking" step. By the time we need to link node X, we've already finished recursing into X's subtrees, so there's no conflict in overwriting its pointers.

04 · Real-world analogy

Threading a necklace bead by bead

Think of the BST nodes as unstrung beads. In-order traversal hands them to you one at a time, in ascending order. You have two hands: one holds the previous bead (prev), the other reaches for the new one (curr). You connect the two beads with a double-ended clasp — that's the two pointer assignments. At the end, you close the necklace by clipping the last bead back to the first.

Key insight: You don't need to know where the "next" bead is before connecting the "current" one. You only ever look backward (at prev) and set two pointers. The in-order traversal guarantees the beads arrive in the right order — you just string them as they come.

05 · Understanding the code line by line

What every line is actually doing

The algorithm has four distinct phases. Understanding each phase separately makes the full solution obvious.

def treeToDoublyList(root):
if not root: return None # PHASE 0: edge case — empty tree
 
head = prev = None
# head: anchors the start of the DLL (set once, at leftmost node)
# prev: the last node we have fully linked — slides forward each step
 
def inorder(node):
nonlocal head, prev
if not node: return # base case: null leaf
 
# PHASE 1: go all the way left first
inorder(node.left)
 
# PHASE 2: process current node (the "visit" step)
if prev:
prev.right = node # prev's "next" → current node
node.left = prev # current's "prev" → previous node
else:
head = node # first visit ever → this is the head
prev = node # slide the prev pointer forward
 
# PHASE 3: go right (larger values)
inorder(node.right)
 
inorder(root)
 
# PHASE 4: close the circle — O(1) regardless of tree size
head.left = prev # head's "prev" → tail (the last node visited)
prev.right = head # tail's "next" → head
return head
Why use nonlocal head, prev?
Python nested functions can read outer variables but cannot rebind them without nonlocal. Since both head and prev need to be reassigned (not just mutated), we need nonlocal. An alternative is to use a single-element list like prev = [None] and mutate prev[0] — this avoids nonlocal but is less readable.
Why is it safe to overwrite node.left during the visit step?
By the time we reach the "visit" step for any node, we have already fully finished recursing into its left subtree. inorder(node.left) has returned. We no longer need node.left as a tree pointer — every node in the left subtree has already been processed and linked. Overwriting it with the prev pointer is safe.
Why is it safe to overwrite node.right on the previous node?
When we set prev.right = node, we are overwriting the right child pointer of prev. But prev has already been visited — we finished its right subtree recursion, returned, and are now at a higher node. prev.right was the link that led us up to the current context, but since recursion has already unwound from it, we no longer need it as a tree pointer.
Can this be solved iteratively?
Yes. Use an explicit stack to simulate in-order traversal. Push nodes onto the stack going left, then pop, link, and go right. The same two-pointer logic (head and prev) applies. This avoids Python's recursion limit on deep trees and achieves the same O(n) time with O(h) stack space.

07 · Closing the circle — final state

The two pointer assignments that make it circular

Don't forget the circular step! Forgetting head.left = prev and prev.right = head at the end is the most common bug. The traversal only builds a linear DLL — you must manually close the circle after the DFS returns.

08 · Common mistakes

Errors to avoid in interviews

✕

Forgetting to close the circle at the end

The traversal builds a linear DLL from head to tail. Without head.left = prev and prev.right = head, the list is not circular. This is the single most common error on this problem.

✕

Using pre-order or post-order instead of in-order

Only in-order traversal visits BST nodes in sorted ascending order. Pre-order visits the root first (unsorted). Post-order visits it last (also unsorted). Replacing inorder(left) / visit / inorder(right) with any other arrangement breaks the sorted property.

✕

Overwriting node.right before recursing right

If you set prev.right = node and then immediately try to recurse inorder(node.right), you are recursing into what is now the linked DLL node — not the BST's right child. Always recurse inorder(node.right) after the linking step, on the originally stored right child. In the code above, the local variable node still holds the original right child reference, so inorder(node.right) at the end is safe — but be careful if you refactor.

✕

Not handling the single-node edge case

For a single-node BST, inorder sets head = prev = that node. Then head.left = prev and prev.right = head both point the node to itself — a valid circular DLL of size 1. The code handles this correctly without a special case, but verify it mentally before your interview.

09 · Complexity — the full explanation

Why O(n) time and O(h) space?

Time
O(n)
Every node is visited exactly once in the in-order traversal. At each visit, exactly two pointer assignments are made (O(1)). The final circle-closing step is O(1). Total: O(n).
Space
O(h)
Only the recursion call stack is used — no extra data structures. Stack depth equals tree height h: O(log n) balanced, O(n) worst case (skewed tree). The conversion itself is in-place: O(1) extra for the two pointers head and prev.

Interview tip: Emphasise "in-place" — no new nodes, O(1) extra space beyond the call stack. If asked about the iterative approach: same O(n) time, same O(h) space using an explicit stack, but avoids Python's recursion limit on trees with depth > ~1000.

10 · Interview Q&A

Questions you'll actually get asked

Why does in-order traversal produce sorted output for a BST?
By the BST property, for any node X: all values in X's left subtree are smaller than X, and all values in X's right subtree are larger. In-order DFS visits the left subtree entirely first (all smaller values), then X, then the right subtree (all larger values). This ordering is recursive — it applies at every subtree — so the full traversal produces values in globally sorted ascending order.
Can you solve this without recursion?
Yes. Use an explicit stack to simulate in-order traversal: push nodes going left until None, pop and visit (link), then go right. Same head and prev pointers. Same O(n) time, O(h) stack space. The iterative version is safer in production when tree depth is unknown and avoids Python's default ~1000 recursion limit.
What if the input has duplicate values?
Duplicates are handled transparently. In-order traversal will still visit all nodes in non-decreasing order. The linking logic does not depend on values being distinct — it only cares about visit order. The resulting DLL will correctly reflect the sorted (with duplicates) sequence.
How would you verify the output is a valid circular DLL?
Starting from head, walk forward via .right until you loop back to head — the values should be in ascending order and you should visit exactly n nodes. Then walk backward via .left from head — you should reach the tail immediately (head.left == tail), and walking backward from tail should bring you back through all nodes in descending order. Also verify tail.right == head.
