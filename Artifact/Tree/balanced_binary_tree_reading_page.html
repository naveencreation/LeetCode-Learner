Balanced Binary Tree

Determine if a binary tree is height-balanced — every node's left and right subtree heights differ by at most 1.

DFS · Recursion Bottom-up LeetCode #110 · Easy

Key concepts at a glance — for those who already know the basics.

01 · The definition

For every node: |height(left) − height(right)| ≤ 1

This condition must hold at every node — not just the root. A tree that looks balanced at the top can still be unbalanced deeper down.

02 · The key insight

1
Use bottom-up DFS to compute height and check balance simultaneously
Post-order traversal: left → right → node. Each node returns its height upward.
2
Return −1 as a sentinel the moment any subtree is unbalanced
Once −1 is returned, every ancestor propagates it immediately — early exit with no wasted work.
3
At the root, if the height is ≥ 0 the tree is balanced
A valid height means every node passed its own balance check during the traversal.

03 · Code (Python)

class Solution:
  def isBalanced(self, root) -> bool:
 
    def check(node):
      if not node: return 0               # null leaf has height 0
 
      left = check(node.left)
      if left == -1: return -1       # early exit
 
      right = check(node.right)
      if right == -1: return -1      # early exit
 
      if abs(left - right) > 1: return -1 # unbalanced here
      return 1 + max(left, right)      # balanced — return real height
 
    return check(root) != -1

06 · Complexity

Time
O(n)
Each node is visited exactly once in the bottom-up DFS. Early exit via −1 sentinel means unbalanced trees terminate even faster.
Space
O(h)
Recursion stack depth equals the tree height. O(log n) for balanced trees, O(n) worst-case for a skewed tree.

01 · What does "balanced" actually mean?

Not just the root — every single node

A common beginner mistake is to only check the root's subtree heights. But the problem requires every node in the entire tree to satisfy the height condition. A tree can look fine at the root and be wildly skewed three levels down.

The formal definition: a binary tree is height-balanced if, for every node, the absolute difference between the height of its left subtree and the height of its right subtree is at most 1. Height of a null subtree is 0.

Height is defined as the number of nodes on the longest path from a node down to a leaf. A leaf node has height 1. A null pointer has height 0.

02 · Why bottom-up, not top-down?

Two approaches — one is dramatically better

Top-down DFS — O(n²)

At every node, compute the left height and right height by traversing again from scratch. Redundantly re-traverses the same subtrees — O(n) per node, O(n²) total.

Bottom-up DFS ← O(n)

Each node returns its own height as it unwinds from recursion. The parent reads these directly — no repeated work. One pass through the tree.

Interview tip: If you mention the naive top-down O(n²) solution, immediately follow with: "We can improve to O(n) using a bottom-up DFS that calculates height and checks balance in the same recursive pass, using a sentinel value of −1 to signal early failure."

03 · The real-world analogy

Think of it like a building inspection

Imagine inspecting a building floor by floor, bottom to top. Each floor inspector checks both wings of that floor and reports back the height of the taller wing plus 1. If any wing already failed inspection, the report is just "FAILED" — and every floor above automatically inherits that failure without doing any more work. The balanced binary tree check works exactly the same way, with subtrees as wings and heights as the report.

Key insight: The −1 sentinel is genius. Instead of using a boolean flag, we encode two pieces of information in a single integer return value: if ≥ 0, it's the actual height; if −1, the subtree is unbalanced. This eliminates the need for a global variable or a separate data structure.

04 · Understanding the code line by line

What the code is actually doing

The entire solution hinges on the inner helper check(node). It is a post-order DFS — it finishes exploring both children before doing anything at the current node. This is critical: we need the children's heights before we can verify the current node's balance.

def check(node):
  if not node: return 0    # base case: null → height 0
 
  left = check(node.left)
  if left == -1: return -1  # left subtree already failed
 
  right = check(node.right)
  if right == -1: return -1  # right subtree already failed
 
  if abs(left - right) > 1:
    return -1                    # this node is the unbalanced point
 
  return 1 + max(left, right)  # balanced here — report actual height
Why do we check left == -1 before even computing the right subtree?
Pure optimization — short-circuit evaluation. If the left subtree is already unbalanced, there's no need to explore the right subtree at all. The tree is guaranteed to be unbalanced regardless of what the right side looks like. This can save O(n/2) work in the worst case.
Why is −1 a safe sentinel? What if a real height is −1?
It's impossible for a real height to be −1. Heights are always ≥ 0 (null = 0, leaf = 1, and each level adds +1). So −1 is permanently outside the range of valid heights, making it a perfect unambiguous sentinel value.
What is post-order traversal and why is it used here?
Post-order means: process left subtree, then right subtree, then the current node. We use it here because a node can only check its own balance after knowing the heights of both children. If we processed the current node first (pre-order), those heights wouldn't exist yet.

Same as above. Returns 1. Both leaves of node-3 are done.

Return

Return

check(3) [right child of left-2] → 1

left=0, right=0. Returns 1. This 3-node has no children.

Fail!

check(2) [left child of root] → −1

left=2, right=1. |2−1|=1 ≤ 1... wait — this node is actually balanced! Returns 1+max(2,1)=3.

Return

Correcting: left subtree of root has height 3. Now let's check the right side.

Return

check(2) [right child of root] → 1

This 2-node is a leaf (no children in this example). left=0, right=0. Returns 1.

left=3, right=1. |3−1|=2 > 1. Root itself is the unbalanced node. Returns −1. Outer function sees −1 ≠ 0, returns false.

06 · Common beginner mistakes

What trips people up

✕

Only checking the root's subtree heights

The root might have |left − right| = 1 but a deeply nested node could have a difference of 3. Every node must be checked, not just the root.

✕

Using top-down DFS and computing height repeatedly

Calling a separate height() function at every node leads to O(n²) time because the same subtrees are traversed over and over. Always combine the height calculation with the balance check in a single pass.

✕

Forgetting to propagate the −1 sentinel upward

After a child returns −1, you must return −1 immediately without doing any further work. Forgetting the early-exit check means you compute nonsense heights on top of a known failure — which can even accidentally return a valid-looking number.

✕

Confusing "balanced" with "complete" or "perfect"

A balanced tree is not necessarily full, complete, or perfect. It simply requires height differences ≤ 1 at every node. A tree can have missing leaves and still be balanced as long as no node violates the condition.

07 · Complexity — the full explanation

Why O(n) time and O(h) space?

Time
O(n)
Each of the n nodes is entered and exited exactly once in the DFS. Once a −1 is produced, it propagates up in O(1) per ancestor — there's no extra work.
Space
O(h)
The call stack holds one frame per level of depth. O(log n) for a balanced tree, O(n) for a completely skewed tree (essentially a linked list).

Interview tip: The space complexity is O(h), not O(n). Distinguish these clearly. For a balanced tree h = O(log n), making the space O(log n). Say: "Space is O(h) for the recursion stack — O(log n) best case for a balanced tree, O(n) worst case for a skewed tree."
