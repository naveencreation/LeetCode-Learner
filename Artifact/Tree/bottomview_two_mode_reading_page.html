Bottom View of Binary Tree

The last node visible at each horizontal distance when the tree is seen from the bottom.

BFS + HashMap Horizontal Distance Interview Essential

Key concepts at a glance — for those who already know the basics.

01 · The rule

Last node seen at each horizontal distance from the root

Assign root HD=0. Left child gets HD−1, right child HD+1. For each HD, the deepest node wins — if two nodes share the same HD and depth, the one processed later (right-wards in BFS) wins.

02 · How to think

1
Do a BFS, tracking each node's horizontal distance (HD)
Root starts at HD = 0. Left child: HD − 1. Right child: HD + 1.
2
For every HD, keep overwriting the map with the current node
BFS visits nodes top-to-bottom, so later overwrites = deeper nodes. That's the bottom view.
3
Read the map sorted by HD from min to max
Gives you the bottom view left-to-right as it would appear from underneath the tree.

03 · Code

from collections import deque
 
def bottom_view(root):
    if not root: return []
    hd_map = {}                        # hd → node value
    queue = deque([(root, 0)])        # (node, hd)
    while queue:
        node, hd = queue.popleft()
        hd_map[hd] = node.val          # always overwrite → last = deepest
        if node.left: queue.append((node.left, hd - 1))
        if node.right: queue.append((node.right, hd + 1))
    return [hd_map[k] for k in sorted(hd_map)]

06 · Complexity

Time
O(n log n)
BFS visits every node once — O(n). Sorting the HD keys at the end adds O(n log n) in the worst case.
Space
O(n)
The queue holds at most the widest level (O(w)). The HashMap stores one entry per unique HD, bounded by n.

01 · What does "bottom view" actually mean?

Seeing the tree from underneath

Imagine you're lying flat on the ground looking up at a binary tree. For every vertical column, you can only see the lowest-hanging node — everything above it is hidden. That's the bottom view: the last (deepest) node at every horizontal position.

The key concept is Horizontal Distance (HD). Root is at HD = 0. Going left subtracts 1; going right adds 1. Two nodes that share the same HD fall in the same vertical column — only the one at the greatest depth (or rightmost if tied) is visible from below.

02 · Why BFS, not DFS?

Two approaches — one is much cleaner

BFS ← natural fit

Always overwrites earlier entries. Because BFS is level-by-level, a later write is always deeper. No depth tracking needed.

DFS (recursive)

Must track depth explicitly and only overwrite when the new depth is ≥ stored depth. More error-prone.

Interview tip: Always lead with BFS. If pressed for a DFS alternative, say: "I'd track (hd, depth) in the map and only overwrite when the new node's depth ≥ the stored depth."

03 · The real-world analogy

Think of it like shadows on the ground

Hang the tree from the ceiling with the root at top. Shine a light directly downward. Each node casts a shadow on the floor. Where two nodes share the same vertical column, the lower node's shadow completely covers the upper one's. The bottom view is precisely the set of nodes whose shadows reach the floor unobstructed — one per column.

Key insight: The number of nodes in the bottom view equals the number of distinct horizontal distances in the tree. Unlike the left/right view (which is always exactly height+1 nodes), the bottom view size depends on the tree's width and structure.

04 · Understanding the code line by line

What the code is actually doing

The trick is deceptively simple: store each node's HD alongside it in the queue. As we dequeue nodes, we unconditionally overwrite hd_map[hd]. Because BFS processes nodes level by level (top to bottom), a later overwrite is guaranteed to be from a deeper level — which is exactly what the bottom view needs.

from collections import deque
 
def bottom_view(root):
    if not root: return []
    hd_map = {}
    queue = deque([(root, 0)])
    while queue:
        node, hd = queue.popleft()
        hd_map[hd] = node.val # always overwrite — deeper = bottom
        if node.left: queue.append((node.left, hd - 1))
        if node.right: queue.append((node.right, hd + 1))
    return [hd_map[k] for k in sorted(hd_map)]
Why do we always overwrite hd_map[hd] without checking depth?
BFS processes nodes level-by-level, which means any node processed after a previous one at the same HD is necessarily at a greater or equal depth. Unconditional overwrite is correct — you never risk overwriting a deeper node with a shallower one when using BFS.
What happens when two nodes share the same HD and same depth?
If two nodes are at the same depth and HD (like nodes 5 and 6 both at HD=0, depth=2 in our example), the one encountered last in BFS order wins — which is the right-side node (since we always enqueue left then right). This is the standard convention for the bottom view.
Why sort the map keys at the end?
A Python dictionary preserves insertion order, but HD values can be inserted in any order during BFS (depending on tree shape). To guarantee left-to-right output (smallest HD first), we must sort the keys explicitly. This adds O(n log n) to the overall time complexity.

Walking through the example tree

Tree: root=1 (HD=0). Left=2 (HD=−1) with children 4 (HD=−2) and 5 (HD=0). Right=3 (HD=+1) with children 6 (HD=0) and 7 (HD=+2). Let's trace every BFS action.

Start

Queue = [(1, 0)]

Root enters the queue with HD = 0.

Write

Dequeue (1, HD=0) → hd_map = {0: 1}

First node at HD=0. Enqueue children: (2, −1) and (3, +1).

Write

Dequeue (2, HD=−1) → hd_map = {0:1, −1:2}

New HD column. Enqueue (4, −2) and (5, 0).

Write

Dequeue (3, HD=+1) → hd_map = {…, +1:3}

New HD column. Enqueue (6, 0) and (7, +2).

Write

Dequeue (4, HD=−2) → hd_map = {…, −2:4}

New HD column. Leaf — nothing enqueued.

Overwrite

Dequeue (5, HD=0) → hd_map[0] = 5 (was 1)

HD=0 already has node 1 (root). Node 5 is deeper, so it overwrites. Leaf — nothing enqueued.

Overwrite

Dequeue (6, HD=0) → hd_map[0] = 6 (was 5)

Same HD=0, same depth as 5 but processed after (BFS right-side bias). 6 wins. Leaf — nothing enqueued.

Write

Dequeue (7, HD=+2) → hd_map = {−2:4, −1:2, 0:6, +1:3, +2:7}

Queue empty. Sort keys → Bottom View = [4, 2, 6, 3, 7] ✓

06 · Common beginner mistakes

What trips people up

✕

Using DFS and forgetting to check depth before overwriting

In DFS, traversal order is not depth-ordered. You must explicitly store the depth alongside the value and only overwrite when new_depth >= stored_depth. Unconditional overwrite in DFS produces wrong answers.

✕

Forgetting to sort the map keys before returning

Insertion order in the map depends on BFS traversal, not HD order. Without sorted(hd_map), the result may be scrambled. Always sort by HD to get a left-to-right answer.

✕

Confusing Bottom View with Leaf nodes

Bottom view nodes are not necessarily leaves. If a non-leaf node is the deepest at its HD column, it still appears in the bottom view. Example: if node 2 (HD=−1) has no children, it appears in the bottom view even though it's an internal node.

✕

Using a regular list as queue (pop from front)

list.pop(0) shifts every element — O(n) per dequeue. Always use collections.deque for O(1) popleft().

07 · Complexity — the full explanation

Why O(n log n) time and O(n) space?

Time
O(n log n)
BFS is O(n) — each node enqueued and dequeued once. Sorting HD keys at the end adds O(n log n). If sorted output isn't required, the algorithm is O(n).
Space
O(n)
The queue holds at most O(w) nodes (widest level). The HashMap has at most n entries (one per node at worst). Both bounded by O(n).

Interview tip: Mention both components of time complexity: "O(n) for the BFS traversal plus O(n log n) for sorting HD keys — dominated by the sort. If the interviewer asks for O(n), you can use a sorted insertion structure like a TreeMap (Java) or SortedDict (Python's sortedcontainers) to avoid the final sort."

08 · Interview context

Where you'll actually use this

Top View of Binary Tree

Identical setup — track HD with BFS — but record only the first node seen at each HD (not the last). One character change: don't overwrite if HD already in map.

Vertical Order Traversal (LeetCode 987)

Same HD tracking but collect all nodes per column, then sort by (row, value). The bottom view is a special case of vertical order.

Width of Binary Tree at each level

The HD range (max HD − min HD + 1) at any given level is the width of that level. Reuse the same BFS + HD setup.

Diagonal Traversal of Binary Tree

Uses a similar column-mapping idea but defines diagonals differently (right edges stay same diagonal, left edges move to next). Related conceptually.
