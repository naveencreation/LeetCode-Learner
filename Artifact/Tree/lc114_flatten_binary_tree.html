Flatten Binary Tree to Linked List

Given the root of a binary tree, flatten it into a "linked list" in-place using the same TreeNode structure, following preorder traversal order.

Preorder DFS Morris Traversal In-place Medium

Key concepts at a glance — for those who already know the basics.

01 · The core insight

For every node: find the rightmost node of its left subtree, stitch it to the node's right child, then swing the entire left subtree to the right and clear the left pointer.

Repeat this locally at each node (iteratively, top-down) — no extra memory, no recursion needed.

02 · How to think (3 steps, repeated for each node)

1
If the current node has a left child, find its rightmost descendant
Walk right as far as possible within the left subtree. That node is the "tail" of the left subtree in preorder.
2
Connect: tail.right = node.right
The right subtree gets appended after the entire left subtree. Nothing is lost.
3
Move left subtree to right: node.right = node.left, node.left = None
The left subtree now occupies the right pointer. Left is cleared. Advance to node.right and repeat.

03 · Code (Python — O(1) space, iterative)

class Solution:
def flatten(self, root) -> None:
curr = root
while curr:
if curr.left:
# 1. Find rightmost node of left subtree
tail = curr.left
while tail.right:
tail = tail.right
 
# 2. Stitch: tail → curr's right subtree
tail.right = curr.right
 
# 3. Move left subtree to right, clear left
curr.right = curr.left
curr.left = None
 
curr = curr.right # advance

01 · Problem in plain English

What exactly are we being asked to do?

We start with a regular binary tree. We need to rearrange its nodes in-place — using the same TreeNode objects, not creating new ones — so that the result looks like a singly linked list threaded through the right pointer of every node. Every left pointer must be set to null.

The order of nodes in the final list must match a preorder traversal of the original tree: Root → Left → Right.

This sequence becomes the required order of the final linked list. Every node's right pointer points to the next element in this sequence. Every left pointer is null.

In-place constraint: The problem says "do not return anything" and "modify the tree in-place." This rules out building a new list from a preorder traversal array. We must rewire the actual pointers of the existing nodes.

02 · The key mechanism — Morris-style rewiring

Why one node's left subtree can be "stitched" in O(1)

The core observation: in preorder, after visiting a node curr, you visit its entire left subtree before visiting its right subtree. So the last node you visit in the left subtree (its rightmost descendant) is the one that should directly precede the right subtree in the final list.

This gives us a purely local operation at curr — no recursion needed:

The stitch operation at node 1

Before

curr = 1
Left subtree rooted at 2, with children 3 and 4.

tail = 4 — rightmost node of the left subtree (node 4 has no right child).

The dashed line shows what we are about to create: tail.right = curr.right.

After (at node 1)

tail.right = curr.right → node 4's right now points to node 5.

curr.right = curr.left → node 1's right now points to node 2.

curr.left = None → node 1 has no left child.

Now advance: curr = curr.right → move to node 2 and repeat.

Order matters: You must stitch tail.right = curr.right before doing curr.right = curr.left. If you do it in reverse, you overwrite curr.right and lose the original right subtree forever.

04 · Amortised analysis — why time is still O(n)

The "find tail" inner loop looks expensive. Is it?

At first glance it seems like the inner while tail.right loop could make this O(n²). But consider: each edge in the original tree is traversed by the inner loop at most once. Once a node is stitched and curr advances past it, the inner loop never visits that segment again. The total number of inner-loop steps across all outer-loop iterations is therefore bounded by the total number of edges, which is n−1. So the algorithm is O(n) time, O(1) space.

Contrast with the naive approach

❌ Collect then rewire

Preorder traversal → list → rewire pointers. O(n) time, but O(n) space for the intermediate list.

✓ Morris-style in-place

No auxiliary data structure. Rewires pointers on the fly. O(1) extra space while remaining O(n) time.

05 · Three ways to solve it

Iterative (best), Recursive, and List-first

① Iterative Morris-style — O(n) time, O(1) space ← preferred

def flatten(self, root) -> None:
curr = root
while curr:
if curr.left:
tail = curr.left
while tail.right: tail = tail.right
tail.right = curr.right
curr.right = curr.left
curr.left = None
curr = curr.right

② Recursive post-order — O(n) time, O(h) space (stack frames)

def flatten(self, root) -> None:
def dfs(node):
if not node: return None
left_tail = dfs(node.left)
right_tail = dfs(node.right)
if left_tail:
left_tail.right = node.right
node.right = node.left
node.left = None
return right_tail or left_tail or node
dfs(root)

③ Preorder list then rewire — O(n) time, O(n) space (simplest to understand)

def flatten(self, root) -> None:
nodes = []
def preorder(node):
if node: nodes.append(node); preorder(node.left); preorder(node.right)
preorder(root)
for i in range(len(nodes) - 1):
nodes[i].right = nodes[i+1]
nodes[i].left = None

06 · Common mistakes

Where people go wrong

✕

Setting curr.right = curr.left before saving the original right

This loses the right subtree. Always do tail.right = curr.right first, then reassign curr.right.

✕

Forgetting to clear curr.left = None

The problem requires all left pointers to be null. If you skip this step the structure is not a valid linked list, and the judge will reject it.

✕

Using an explicit stack and calling it "O(1) space"

A stack-based iterative DFS is still O(h) space. Only the Morris/pointer-rewiring approach achieves true O(1) extra space.

✕

Thinking the inner "find tail" loop makes this O(n²)

Amortised analysis shows the total inner-loop work is O(n). Each edge is traversed by the inner loop at most once across the entire run.

07 · Complexity

Time and space — detailed

Time
O(n)
Each of the n nodes is visited by curr once. The inner "find tail" loop's total steps across all iterations are bounded by n−1 (total edges). Combined: O(n).
Space
O(1)
Only pointer variables (curr, tail). No stack, no recursion, no auxiliary array. This is the primary advantage of the Morris-style approach over the recursive solution.

Interview tip: Lead with the O(n) / O(1) iterative solution. Then mention the recursive alternative (O(h) space) and the preorder-list approach (O(n) space, easiest to reason about). Explaining the trade-offs clearly is exactly what interviewers reward.

08 · Interview Q&A

Questions you'll actually get asked

Why preorder and not inorder or postorder?
The problem explicitly says "flatten the tree as if it were traversed using preorder." Preorder (Root → Left → Right) is the natural reading order of a tree from top-left to bottom-right. The problem could have asked for inorder or postorder — the in-place rewiring technique works for those too, but with a different traversal order driving the stitching logic.
Can you solve this recursively? What's the space cost?
Yes — the recursive approach processes subtrees and returns the tail (rightmost node) of each. It's elegant and easy to reason about, but uses O(h) stack frames, where h is the tree height. In the worst case (completely skewed tree), that's O(n) space — worse than the iterative approach.
What if the tree is already a right-skewed chain?
Then every node has no left child, and the if curr.left check fails every iteration. The algorithm simply walks the chain in O(n) with zero stitching operations. It handles this edge case naturally without special-casing.
What if root is null?
curr = root sets curr to null immediately, so the while loop never executes. The function returns without doing anything, which is the correct behaviour for an empty tree.
