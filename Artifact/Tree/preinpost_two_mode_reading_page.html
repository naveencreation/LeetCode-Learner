Binary Tree · DFS

Pre, In & Post Order in One Traversal

Compute all three DFS traversals simultaneously in a single pass using an explicit stack and a visit-count per node.

Stack DFS Optimization Interview Hard

Key concepts at a glance — for those who already know the basics.

01 · The core idea

Each node is pushed onto the stack with a counter starting at 1.
State 1 → record Preorder, push left
State 2 → record Inorder, push right
State 3 → record Postorder, pop

Every node is visited exactly 3 times. The state counter decides what to do on each visit.

02 · The three traversals compared

Preorder

Root → L → R

Result: [1,2,4,5,3,6,7]

Inorder

L → Root → R

Result: [4,2,5,1,6,3,7]

Postorder

L → R → Root

Result: [4,5,2,6,7,3,1]

Normally these need 3 separate DFS passes. This algorithm does all three in one pass.

03 · Algorithm steps

1
Push (root, state=1) onto the stack
Each stack frame is a (node, count) pair.
2
Pop top frame. Check the state counter.
Increment counter and push the frame back before doing any child push.
3
State 1 → add to pre[], push left child (state=1)
4
State 2 → add to in[], push right child (state=1)
5
State 3 → add to post[], do not push back
Node is fully processed. Naturally falls off the stack.

04 · Code

def allTraversals(root):
pre, ino, post = [], [], []
stack = [(root, 1)] # (node, state)
while stack:
node, state = stack.pop()
if state == 1: # first visit → preorder
pre.append(node.val)
stack.append((node, 2)) # come back at state 2
if node.left: stack.append((node.left, 1))
elif state == 2: # second visit → inorder
ino.append(node.val)
stack.append((node, 3)) # come back at state 3
if node.right: stack.append((node.right, 1))
else: # third visit → postorder
post.append(node.val)
return pre, ino, post

05 · Complexity

Time
O(n)
Each node is pushed/popped exactly 3 times. Total operations = 3n.
Space
O(n)
Stack holds at most O(h) frames at once. Output arrays together hold 3n values.

06 · Interview uses

Optimize multiple traversal calls — when a problem requires more than one traversal result, this avoids repeated tree walks.

Iterative DFS mastery — demonstrates deep understanding of the call stack and how recursion works under the hood.

Tree reconstruction — preorder + inorder together uniquely identify a binary tree. Getting both in one pass is efficient.

Space-constrained environments — avoids 3× recursion overhead when working with very deep or large trees.

01 · Why do we need this at all?

The problem with three separate traversals

You already know how to do inorder, preorder, and postorder traversal individually — each is a clean recursive function. But what if you need all three results from the same tree? The naive approach is to call three separate functions, each walking every node once. That works, but it means three full passes over the tree.

This algorithm does all three traversals in a single pass — O(n) time, touching each node exactly 3 times instead of n×3 = 3n separate function calls spread across three traversals.

The key insight: when recursion traverses a tree, every node is actually encountered three times — once going in (before left), once between left and right, and once coming back up (after right). Those three moments are exactly when preorder, inorder, and postorder record their values.

02 · The three moments of a node

When does each traversal "see" a node?

Think about what happens when the recursive call stack visits any node during a normal DFS. It passes through that node three distinct times:

Visit
When it happens
Which traversal records here
1st
Just entered the node, before going left
Preorder — Root before children
2nd
Returned from left subtree, before going right
Inorder — Root between children
3rd
Returned from right subtree, about to go up
Postorder — Root after children

This is the fundamental insight. All three traversals are just different "checkpoints" at the same node during a single DFS walk. The algorithm makes this explicit with a state counter.

03 · The state counter — the key mechanism

How the stack tracks which visit this is

Since we're implementing this iteratively (without recursion), we need to manually track "which visit is this for this node." We do this by storing a state counter alongside each node on the stack.

Each stack entry is a pair: (node, state) where state starts at 1 and increments each time we revisit the node.

State = 1

First arrival

→ Record for preorder, push left child

State = 2

Back from left

→ Record for inorder, push right child

State = 3

Back from right

→ Record for postorder, done

Why do we push the node back with an incremented state before pushing the child?
Because we want to return to this node after the child is fully processed. The stack is LIFO — last in, first out. So we push the node back first (it will be processed later), then push the child on top (it will be processed next). When the child finishes, the parent node is waiting on the stack at the next state.
What if a node has no left or no right child?
We simply skip the push. If node.left is None at state 1, we don't push anything for the left child — we just push the node back at state 2. The algorithm handles missing children naturally with the if node.left: guard.
What does the stack look like at its deepest point?
When we first reach node 4 (leftmost leaf) from root 1, the stack looks like this:
→ (node=4, state=1)  ← about to process
   (node=2, state=2)  ← waiting for left to finish
   (node=1, state=2)  ← waiting for left to finish

04 · Code — line by line

Understanding every line

def allTraversals(root):
pre, ino, post = [], [], []
stack = [(root, 1)] # start: root at state 1
while stack: # keep going until stack empty
node, state = stack.pop()
if state == 1:
pre.append(node.val) # 1st visit = preorder
stack.append((node, 2)) # come back at state 2
if node.left: # push left child if exists
stack.append((node.left, 1))
elif state == 2:
ino.append(node.val) # 2nd visit = inorder
stack.append((node, 3)) # come back at state 3
if node.right: # push right child if exists
stack.append((node.right, 1))
else: # 3rd visit = postorder, done
post.append(node.val)
return pre, ino, post

Key detail: At state 3, we do NOT push the node back. It simply exits the loop. This is what makes the node "fully done" — no more revisits scheduled.

06 · Common beginner mistakes

What trips people up

✕

Forgetting to push the node back before pushing the child

At state 1, you must push (node, 2) before pushing the left child. If you push the child first, the parent will be processed immediately after the child is pushed — before the child's subtree is done.

✕

Pushing the right child at state 1 instead of state 2

The right child must only be pushed at state 2 — after the left subtree is completely done. Pushing it at state 1 causes the right subtree to be traversed before the left, breaking all three orders.

✕

Pushing the node back at state 3

At state 3, the node is done — do not push it back. Doing so creates an infinite loop. The node only records for postorder and exits.

✕

Using recursion and thinking this algorithm applies

This state-counter trick is specific to the iterative approach. In recursion, the call stack does the state tracking implicitly — you can't attach a counter to a recursive call the same way.

07 · Complexity — the full explanation

Why O(n) time and O(n) space?

Time
O(n)
Each of the n nodes is pushed and popped exactly 3 times — for states 1, 2, and 3. Total stack operations = 3n = O(n).
Space
O(n)
Stack holds O(h) frames at once (h = tree height). The three output arrays hold n values each = 3n total. Combined: O(n).

Interview tip: Compared to three separate recursive traversals (each O(n) time, O(h) space), this algorithm uses the same asymptotic complexity but better constant factors — one tree walk, one stack, no repeated traversal overhead. Mention this trade-off explicitly.

Where you'll actually use this

Tree reconstruction from traversals

You need both preorder and inorder to reconstruct a unique binary tree. Getting both in one pass is more efficient than two separate traversals.

Iterative DFS — signal of advanced understanding

Implementing this iteratively shows you understand what the call stack is doing during recursion — a strong signal in senior-level interviews.

Serialize and deserialize binary tree

Collecting preorder + inorder in one pass can feed directly into a deserialization algorithm without a second traversal.

Understanding all three orders simultaneously

If asked "explain how preorder, inorder, and postorder differ," this algorithm is the clearest way to demonstrate it — same tree, same walk, just different checkpoints.
