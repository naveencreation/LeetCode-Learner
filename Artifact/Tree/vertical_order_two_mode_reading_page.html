Binary Tree · BFS + Sorting

Vertical Order Traversal

Group all nodes by column (horizontal distance), then by row, then by value — and return each column's nodes top-to-bottom.

BFS Sorting HashMap LeetCode 987

Key concepts at a glance — for those who already know the basics.

01 · The rule

Assign every node a (col, row) coordinate.
Group by col → sort each group by (row, value) → return columns left to right.

Root is at (col=0, row=0). Left child: col−1. Right child: col+1. Each level down: row+1.

02 · The key difference from bottom/top view

Bottom / Top View

One node per column. Either the topmost or bottommost wins.

Vertical Order Traversal

All nodes per column, sorted by row then value. No node is dropped.

03 · How to think

1
BFS the tree, tracking (node, col, row) for each node
Root starts at (col=0, row=0). Left: col−1, row+1. Right: col+1, row+1.
2
Collect into a map: col → list of (row, val)
Every node appends to its column's list — nothing is overwritten or discarded.
3
Sort each column's list by (row, val)
Nodes in the same cell (same col and row) are ordered by value ascending.
4
Return columns sorted by col from min to max
Extract just the values from each sorted column list.

05 · Code

from collections import defaultdict, deque
 
def verticalOrder(root):
if not root: return []
col_map = defaultdict(list) # col → [(row, val), ...]
queue = deque([(root, 0, 0)]) # (node, col, row)
while queue:
node, col, row = queue.popleft()
col_map[col].append((row, node.val)) # collect all nodes
if node.left:
queue.append((node.left, col-1, row+1))
if node.right:
queue.append((node.right, col+1, row+1))
result = []
for col in sorted(col_map): # columns left → right
sorted_col = sorted(col_map[col]) # sort by (row, val)
result.append([v for _, v in sorted_col])
return result

Space

01 · What does "vertical order" actually mean?

Imagining the tree as a grid

Picture the binary tree drawn on graph paper. Every node sits at an exact grid position — a column and a row. The root is at (col=0, row=0). Each step left decreases the column by 1, each step right increases it by 1. Each level down increases the row by 1.

Vertical order collects all nodes in each column. Unlike top/bottom view which keeps only one node per column, vertical order keeps every node and groups them by (row, value). The orange nodes 5 and 6 both land on col=0 at row=2 — both appear in the output, sorted by value.

02 · Why this is harder than top/bottom view

Three things that make vertical order tricky

#
The challenge
1
Multiple nodes per column. You can't just overwrite — you must collect every node and store it alongside its row number.
2
Tie-breaking by value. Two nodes can share the same (col, row). In that case, the smaller value comes first. This is a common interview trap.
3
Sorting at the end. Within each column the nodes must be in (row, val) order — BFS alone doesn't guarantee this when ties exist.

The most common mistake in interviews: forgetting to sort by value when two nodes share the same (col, row). BFS gives them in arrival order which may not be value order.

03 · Why BFS with (col, row) tracking

Why not DFS?

You can use either BFS or DFS for this problem. Here is the honest comparison:

BFS — recommended

Naturally processes level by level. Row is implicitly tracked. Simpler to reason about. You still need to sort per column at the end.

DFS — also valid

Must explicitly pass (col, row) through recursion. Same final sort needed. Slightly harder to read. Produces the same result.

Both approaches have the same O(n log n) time complexity because the final sort dominates. BFS is preferred because it's more intuitive for level-based grouping.

04 · The real-world analogy

Think of the tree as a city skyline

Imagine the tree is a set of buildings on a street. Each building stands at a specific street position (column) and has multiple floors (rows). Vertical order traversal is like reading out every building from left to right, and for each building listing every floor from ground up. If two buildings share the exact same address and floor, you list the smaller-numbered one first.

The critical difference from bottom/top view: in those, you only report the ground floor (bottom) or the top floor (top) of each building. In vertical order, you report every floor of every building.

05 · Code — line by line

Understanding every decision in the code

from collections import defaultdict, deque
 
def verticalOrder(root):
if not root: return [] # edge case: empty tree
col_map = defaultdict(list) # col → [(row, val), ...]
queue = deque([(root, 0, 0)]) # (node, col, row)
while queue:
node, col, row = queue.popleft()
col_map[col].append((row, node.val)) # store row too — needed for sort
if node.left:
queue.append((node.left, col-1, row+1))
if node.right:
queue.append((node.right, col+1, row+1))
result = []
for col in sorted(col_map): # left col → right col
sorted_nodes = sorted(col_map[col]) # sort by (row, val) — tuple sort
result.append([v for _, v in sorted_nodes])
return result
Why store (row, val) instead of just val in col_map?
Because within a column, nodes must be sorted top-to-bottom (by row), and ties broken by value. If you only stored val, you'd lose the row information needed to sort correctly. Storing (row, val) as a tuple lets Python's default tuple sort handle both: it first sorts by row, then by val for equal rows.
Does Python's sorted() on a list of tuples sort by the first element then second?
Yes. Python sorts tuples lexicographically — it compares the first element first, and only moves to the second element if the first is equal. So sorted([(2,6),(2,5),(1,3)]) gives [(1,3),(2,5),(2,6)] — which is exactly the (row, val) order we need.
Why use defaultdict(list) instead of a regular dict?
defaultdict(list) auto-creates an empty list for any new key, so col_map[col].append(...) works without checking if the key exists first. With a regular dict you'd need col_map.setdefault(col, []).append(...) — same result, more verbose.

07 · Common beginner mistakes

What trips people up

✕

Overwriting instead of appending to col_map

Vertical order keeps ALL nodes per column. Using col_map[col] = val (overwrite) instead of col_map[col].append((row, val)) loses nodes at the same column and gives wrong output.

✕

Not storing row — only storing value

If you only store col_map[col].append(val), you lose the row information. Without row you cannot correctly sort nodes within the same column top-to-bottom.

✕

Forgetting to sort by value when row is tied

If two nodes share the same (col, row), BFS arrival order is not guaranteed to be value order. sorted(col_map[col]) on tuples handles this automatically — but if you sort only by row, tied nodes may appear in the wrong order.

✕

Forgetting to sort col_map keys before building result

dict keys in Python 3.7+ preserve insertion order, which for BFS is root → left subtree → right subtree — not necessarily left-to-right by column. Always use sorted(col_map).

✕

Confusing this with bottom view

Bottom view keeps only one node per column (the deepest). Vertical order keeps all of them. The code change is small — append vs overwrite — but the output is completely different.

08 · Complexity — the full explanation

Why O(n log n) time and O(n) space?

Time
O(n log n)
BFS visits each node once — O(n). Sorting all n nodes across all columns combined is O(n log n). Sorting the column keys is at most O(n log n) too. Dominated by the sort.
Space
O(n)
Queue holds at most one full level at a time — O(w) where w is max width, bounded by O(n). col_map stores all n (row, val) pairs. Total: O(n).

Interview tip: If asked "can you do this in O(n)?", the answer is no in the general case — the sort is necessary and unavoidable. However, if you know the column range in advance (e.g., bounded width), you could use counting sort for O(n). Mention this to show depth.
