Left View of Binary Tree

The first node visible at each level when the tree is seen from the left side.

BFS / DFS Level Order Interview Essential

Key concepts at a glance — for those who already know the basics.

01 · The rule

First node seen at each level from the left

Equivalently: the first node you encounter per depth in a level-order (BFS) traversal.

02 · How to think

1
Process the tree level by level (BFS with a queue)
Each queue iteration handles exactly one full level.
2
Record the very first node dequeued at each level
That first node is the leftmost visible node at that depth.
3
Add left child first, then right child, to the queue
Ensures left nodes always precede right nodes in the queue.

03 · Code

from collections import deque
 
def left_view(root):
    if not root: return []
    result, queue = [], deque([root])
    while queue:
        level_size = len(queue)           # nodes on this level
        for i in range(level_size):
            node = queue.popleft()
            if i == 0: result.append(node.val) # first = left view
            if node.left:  queue.append(node.left)
            if node.right: queue.append(node.right)
    return result

01 · Start here — what is a "view" of a tree?

Seeing the tree from the side

Imagine standing far to the left of a binary tree and looking directly at it. Some nodes will be hidden behind others. The nodes you can actually see — one per level — are called the Left View. Equivalently, the left view is the set of nodes that are first encountered at each depth when you sweep through the tree level by level.

Blue nodes are visible from the left. Grey nodes are hidden behind a blue node at the same level — they share the same depth but are further right.

02 · Why BFS? Why not DFS?

The problem is fundamentally level-based

The left view asks "which node is first at each level?" — that's exactly what BFS (Breadth-First Search) is designed to answer. BFS processes the tree level by level, so finding the first node per level is trivial. DFS can also solve it (using recursion and a max-depth trick), but BFS maps to the problem more naturally.

BFS ← natural fit

Process level by level. First dequeued node = left view.

DFS (recursive)

Track max depth seen. Record node only if its depth is new.

Interview tip: If the interviewer says "can you do it without a queue?", they want the DFS approach. Be ready to explain both. The DFS approach uses a max-depth variable instead of a queue.

03 · The real-world analogy

Think of it like a theatre seating chart

Imagine a theatre where rows of seats fan out from a central aisle. Every row is a level of the tree. You're standing at the far left entrance, looking down the rows. In each row, you see only the leftmost occupied seat — the rest are blocked. The Left View of a tree is exactly those leftmost seats, one per row.

Key insight: The left view has exactly as many nodes as the height of the tree — one node per level. A tree with height h has exactly h+1 nodes in its left view.

04 · Understanding the BFS approach line by line

What the code is actually doing

The core trick is: at the start of each BFS iteration, we know exactly how many nodes are in the current level — it's len(queue). So we loop that many times, and the very first node dequeued (index 0) is the leftmost node at that level.

from collections import deque
 
def left_view(root):
    if not root: return []
    result, queue = [], deque([root])
    while queue:
        level_size = len(queue)
        for i in range(level_size):
            node = queue.popleft()
            if i == 0:
                result.append(node.val) # leftmost at this level
            if node.left:  queue.append(node.left)
            if node.right: queue.append(node.right)
    return result
Why do we snapshot level_size = len(queue) before the loop?
Because the queue grows as we add children during the loop. If we checked len(queue) dynamically, we'd process nodes from the next level too, mixing levels together. By snapshotting the count before, we know exactly how many nodes belong to the current level.
Why add the left child before the right child?
Order in the queue determines which node is dequeued first. If we added the right child first, the right child would appear before the left child at the next level — and our "first dequeued" would give us the right view, not the left view. Left first → left view. Right first → right view.
What does the queue look like during execution?
Using the example tree (root=1, left=2, right=3, 2's children=4,5, 3's right=7):
Start of L0:  [1]           ← dequeue 1, record it
Start of L1:  [2, 3]         ← dequeue 2 first, record it
Start of L2:  [4, 5, 7]      ← dequeue 4 first, record it

05 · Full trace — every step explained

Walking through the example tree

Tree: root=1, left child=2 (children: 4, 5), right child=3 (right child: 7). Let's trace every BFS action.

L0 Start

Queue = [1], level_size = 1

Only the root is in the queue. We're about to process level 0.

Record

Dequeue 1 (i=0) → result = [1]

First node at level 0 — record it. Add its children: left=2, right=3. Queue is now [2, 3].

L1 Start

Queue = [2, 3], level_size = 2

Two nodes at this level. We'll dequeue 2 first (it was added first — left child before right child).

Record

Dequeue 2 (i=0) → result = [1, 2]

First node at level 1 — record it. Add children: 4 and 5. Queue is now [3, 4, 5].

Skip

Dequeue 3 (i=1) → not recorded

Not the first node at level 1, so we skip recording it. Add its right child 7. Queue is now [4, 5, 7].

L2 Start

Queue = [4, 5, 7], level_size = 3

Three nodes at this level. Node 4 (leftmost) will be dequeued first.

Record

Dequeue 4 (i=0) → result = [1, 2, 4]

First node at level 2 — record it. Node 4 is a leaf, nothing added to queue.

Skip

Dequeue 5 (i=1) → not recorded

Not the first at this level. Leaf node, nothing enqueued.

Skip

Dequeue 7 (i=2) → not recorded. Queue empty. Done ✓

Final left view: [1, 2, 4] — one node per level, always the leftmost visible one.

06 · Common beginner mistakes

What trips people up

✕

Not snapshotting level_size before the loop

If you write for i in range(len(queue)) directly, the queue grows mid-loop as you add children, causing you to mix up levels — the next level's nodes bleed into the current iteration.

✕

Adding right child before left child

The order of enqueuing determines which node appears first. Enqueue left before right — otherwise your "first dequeued" gives you the rightmost node, not the leftmost.

✕

Forgetting to handle the empty tree

If root is None, calling deque([None]) and then accessing node.left will crash. Always guard with if not root: return [].

✕

Using a regular list instead of deque

list.pop(0) is O(n) because it shifts every element. deque.popleft() is O(1). For large trees this difference matters — always use collections.deque.

07 · Complexity — the full explanation

Why O(n) time and O(w) space?

Time
O(n)
Every node is enqueued once and dequeued once. Recording the first node per level is an O(1) check. Total: O(n).
Space
O(w)
The queue holds at most the nodes of the widest level. w = 1 for a skewed tree, w = n/2 for a perfect binary tree — so worst case O(n).

Interview tip: When asked about space complexity, clarify: "O(w) where w is the maximum width of the tree. In the worst case of a perfect binary tree the last level holds n/2 nodes, so it's O(n). For a skewed tree it degenerates to O(1) queue space."
