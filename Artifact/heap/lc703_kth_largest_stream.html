Kth Largest Element in a Stream

Design a class that continuously tracks the kth largest element as new values are added to a live data stream — without re-sorting the entire collection each time.

Min-Heap of size k Priority Queue Data Stream Design Easy

Key concepts at a glance — for those who already know the basics.

01 · The core insight

Keep a min-heap of exactly k elements. The heap root is always the kth largest — every element in the heap is larger than every element not in the heap.

When a new value arrives: push it onto the heap. If heap size exceeds k, pop the minimum. The root is then your answer — O(log k) per call, forever.

02 · How to think (3 steps)

1
On init: push all elements from nums into a min-heap, then trim it to size k
Use heapq.heapify() then pop extras. After trimming, heap[0] is already the kth largest of the initial data.
2
On every add(val): push val, then pop if heap size exceeds k
If val is smaller than all k current elements it gets pushed and immediately popped — it never displaces a top-k element.
3
Return self.heap[0] — the heap root is always the answer
The min of the top-k elements is by definition the kth largest across all elements seen so far.

03 · Code (Python)

import heapq
 
class KthLargest:
 
def __init__(self, k: int, nums: list[int]):
self.k = k
self.heap = nums
heapq.heapify(self.heap) # O(n) — build min-heap in-place
while len(self.heap) > k: # trim to exactly k elements
heapq.heappop(self.heap)
 
def add(self, val: int) -> int:
heapq.heappush(self.heap, val) # always push first
if len(self.heap) > self.k: # overflowed k → pop smallest
heapq.heappop(self.heap)
return self.heap[0] # root = kth largest

01 · What the problem is really asking

Streaming data, not a static array

The word "stream" is key. You don't receive all the data up front — elements arrive one at a time, and after each one you must immediately report the kth largest element across everything seen so far. Sorting the entire collection after each addition would work but costs O(n log n) per call, making it totally impractical for high-frequency streams.

The kth largest element is the element that would sit at index k−1 from the right if all elements seen so far were sorted in descending order. Equivalently, it's the smallest element among the top-k largest elements. This equivalence is the key to the heap solution.

Note: kth largest is in sorted order, not kth distinct. Duplicates count — the 2nd largest of [3, 3, 3] is 3, not "no second distinct value."

02 · Why a min-heap, not a max-heap?

The counterintuitive choice explained

When asked "find the kth largest", beginners reach for a max-heap. The logic seems obvious: "I want large elements, so use a max structure." But that's backwards. Here's why a min-heap of size k is the correct choice:

Max-heap — wrong tool here

The max-heap root is the largest element. To find the kth largest you'd need to pop k times — O(k log n) per query, and you'd need to rebuild after each pop. Doesn't help with a stream.

Min-heap of size k ← correct

Keep only the top-k elements. The min of those k is the kth largest. The min-heap gives instant O(1) access to that minimum at the root. Each update is O(log k).

Mental model: Think of the heap as a bouncer at a VIP room with exactly k seats. New arrivals only get in if they're "larger" than the current minimum VIP (the root). When they enter, the smallest VIP is ejected to make room. The person just barely inside (the root) is always the kth largest overall.

04 · Why the initialisation trim matters

What happens when nums has more than k elements

The constructor receives an initial array nums which can have any length. The heap must be trimmed to exactly k elements before the first add() is called. Here's the full initialisation flow:

def __init__(self, k, nums):
self.k = k
self.heap = nums                  # reuse the list in-place
heapq.heapify(self.heap)         # O(n): rearrange into min-heap order
while len(self.heap) > k:       # pop smallest until size = k
heapq.heappop(self.heap)     # each pop: O(log n), done n-k times
Why use heapify then trim, rather than pushing elements one by one?
heapify converts a list into a valid min-heap in O(n) time using a clever bottom-up sifting algorithm. Pushing n elements individually onto an empty heap costs O(n log n). For large initial arrays, the heapify-then-trim approach is meaningfully faster.
What if nums has fewer than k elements?
The trim loop condition len(heap) > k simply never executes — no trimming happens. The heap starts with fewer than k elements and grows toward k as add() is called. The problem guarantees there will always be at least k elements in the stream when you query for the kth, so heap[0] is safe to return once the heap reaches size k.

05 · Three approaches — and why heap wins

Brute force vs sorting vs min-heap

① Brute force sort — O(n log n) per add()

def add(self, val):
self.data.append(val)
self.data.sort(reverse=True)          # O(n log n) every single call
return self.data[self.k - 1]

② Sorted insert — O(n) per add() using bisect

import bisect
def add(self, val):
bisect.insort(self.data, val)     # O(n) shift cost despite O(log n) search
return self.data[-(self.k)]

③ Min-heap of size k — O(log k) per add() ← preferred

def add(self, val):
heapq.heappush(self.heap, val)
if len(self.heap) > self.k:
heapq.heappop(self.heap)
return self.heap[0]             # O(log k), heap stays fixed size k

07 · Common mistakes

Where people go wrong

✕

Using a max-heap instead of a min-heap

A max-heap of all n elements would require popping k times to get the kth largest — O(k log n) per query, and the heap can't stay at fixed size for a stream. The min-heap of size k approach is the correct design.

✕

Checking val > heap[0] before pushing

This skips the push for smaller values but also skips the push when the heap has fewer than k elements — causing it to never reach size k. Always push unconditionally, then pop if size exceeds k.

✕

Forgetting that Python's heapq is a min-heap by default

In Python, heapq always produces a min-heap. You don't need to negate values or pass custom comparators for this problem. In Java, use PriorityQueue<Integer> (default min-heap).

✕

Off-by-one on k vs k−1

The heap holds exactly k elements. heap[0] is the minimum of those k — which is the kth largest. Returning heap[k-1] or keeping k−1 elements are common off-by-one mistakes that produce the (k−1)th or (k+1)th largest.

08 · Complexity — the full breakdown

Time and space — detailed

Init time
O(n log k)
heapify is O(n). Trimming n−k elements costs O((n−k) log n) ≈ O(n log k). Overall initialisation is O(n log k).
add() time
O(log k)
Heap stays at size ≤ k+1. Both heappush and heappop on a heap of size k are O(log k). One O(1) root read at the end.
Space
O(k)
The heap stores exactly k elements at all times after init. No auxiliary data structure. Total space is O(k) regardless of how many elements have been added.
m add() calls
O(m log k)
If add() is called m times, total cost is O(m log k). This is the dominant cost for long-running streams where m ≫ n.

Interview tip: State time complexity separately for init and add(). Say: "Initialisation is O(n log k) where n is the length of nums. Each subsequent add() is O(log k) since the heap stays bounded at size k. Space is O(k)." This level of precision shows you understand the design, not just the code.

09 · Interview Q&A

Questions you'll actually get asked

Why is heap[0] the kth largest, not the kth smallest?
The heap contains the k largest elements seen so far. Among those k, heap[0] is the smallest (it's a min-heap). The smallest among the top-k is by definition the kth largest overall. Anything smaller than heap[0] is not in the top-k and would be popped if added.
What if k=1? What does the heap track?
The heap holds exactly 1 element — the global maximum seen so far. Every time a new maximum arrives it displaces the current one. heap[0] always returns the 1st largest (i.e., the maximum). The algorithm handles k=1 naturally without any special casing.
Can the stream contain duplicates?
Yes — and the heap handles them transparently. Duplicates are stored as separate entries. If the 3rd largest is 5 and the stream sends another 5, both copies can coexist in the heap. Python's heapq compares values numerically and doesn't distinguish between duplicates.
How would you solve this if k could change between calls?
If k is dynamic, you'd need to rebuild the heap each time k changes — O(n log k) to re-trim. Alternatively, maintain a sorted structure (like a balanced BST or a sorted list via sortedcontainers.SortedList in Python) that supports O(log n) insert and O(1) index access. That gives O(log n) per add regardless of k.
