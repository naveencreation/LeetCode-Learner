Kth Largest Element in an Array

Given an unsorted integer array and an integer k, return the k-th largest element in sorted order (duplicates count).

Min-Heap Quickselect Divide & Conquer Medium

Key concepts at a glance — for those who already know the basics.

01 · The rule

Keep a min-heap of exactly k elements.
After scanning all n elements, the heap root is the answer.

The heap always holds the k largest elements seen so far. Its minimum (the root) is the k-th largest overall.

02 · How to think

1
Push each element onto a min-heap
A min-heap always exposes its smallest element at the root in O(1).
2
If the heap size exceeds k, pop the smallest element
This evicts the current weakest contender, keeping only the k strongest candidates.
3
After all elements are processed, return heap[0]
The heap root is now the smallest of the k largest — which is exactly the k-th largest.
4
Alternative: Quickselect gives O(n) average time
Partition around a pivot and recurse only into the side that contains the target rank.

03 · Code — Min-Heap approach (Python)

import heapq
 
class Solution:
def findKthLargest(self, nums, k) -> int:
heap = []
for num in nums:
heapq.heappush(heap, num) # push every element
if len(heap) > k:
heapq.heappop(heap) # discard smallest
return heap[0] # root = kth largest

01 · Start here — what does "k-th largest" actually mean?

Pinning down the definition before writing any code

The k-th largest element is the element that sits at position k when the array is sorted in descending order — where position 1 is the maximum. Crucially, duplicates count as separate positions. If the same value appears twice, it occupies two positions.

Common trap: "k-th largest" is NOT "k-th distinct largest." In the example above the values 5 and 5 are two separate positions. The 4th largest is 4, not 3. Always count duplicates as individual positions.

02 · Three approaches — why not just sort?

From O(n log n) brute force to O(n) optimal

The problem explicitly asks: can you solve it without sorting? There are three progressively better approaches, each with a real trade-off.

Sort

O(n log n)

Sort descending, return index k−1. Simple but wastes work ordering every element.

Min-Heap ✓

O(n log k)

Maintain only k elements. Deterministic, works on streams, O(k) extra space.

Quickselect

O(n) avg

In-place partition. O(1) space, O(n²) worst-case — randomise pivot to mitigate.

Interview guidance: In most interviews the min-heap solution is the expected answer (predictable O(n log k), easy to reason about). Know Quickselect too — interviewers often ask "can you do better?" as a follow-up. The heap is preferred for streaming data; Quickselect wins for offline arrays with small k.

03 · Understanding the min-heap approach

Why the smallest element in the heap gives us the k-th largest overall

The key insight is this: if you always maintain the k largest elements seen so far, the smallest among them is exactly the k-th largest. The min-heap is perfect for this — its root is always the minimum, which is the "weakest" of the k champions we're tracking.

The invariant: After every element is processed, the heap contains exactly the k largest elements seen so far, with the k-th largest (the weakest champion) sitting at the root. This invariant holds for every single step.

04 · Min-Heap code — every line explained

Reading the solution line by line

import heapq # Python's built-in min-heap (smallest at index 0)
 
def findKthLargest(nums, k):
heap = [] # ① empty min-heap
for num in nums: # ② process every element once
heapq.heappush(heap, num) # ③ push unconditionally
if len(heap) > k: # ④ heap grew too large
heapq.heappop(heap) # ⑤ evict the smallest
return heap[0] # ⑥ root = kth largest
③ Why push first unconditionally, then check size?
We need to compare the new element with the current weakest champion (root) to decide whether it earns a spot. Pushing first and then popping the minimum achieves this in one sequence. If the new element is smaller than all k current champions it pops itself right back out — correct behaviour with zero special-casing.
⑤ When we pop, are we always removing the new element?
No — we pop whatever is currently the minimum of the k+1 elements. If the new element is smaller than all k current champions, it becomes the new minimum and pops itself. If it is larger than some champions, a previous element gets popped instead. Either way the k strongest elements remain.
⑥ Why is heap[0] the answer and not the last element?
Python's heapq is a min-heap: the smallest element is always at index 0. After processing all n elements, the heap holds exactly the k largest. The smallest of those — at index 0 — is by definition the k-th largest overall. The other elements in the heap are ranked 1st through (k-1)th.

05 · Full trace — Min-Heap on [3,2,1,5,6,4], k=2

Walking through every push and pop

Push 3

heap = [3], size = 1 ≤ 2 — no pop

First element enters the heap. Size is within limit.

Push 2

heap = [2, 3], size = 2 = k — no pop

Heap now at capacity. Min-heap property: 2 sits at root (index 0) because 2 < 3.

Pop 1

Push 1 → heap = [1, 3, 2] → pop 1 → heap = [2, 3]

Size exceeded k=2. The new element 1 is smaller than both champions — it becomes the minimum and gets evicted immediately. Champions unchanged: {2, 3}.

Pop 2

Push 5 → heap = [2, 3, 5] → pop 2 → heap = [3, 5]

5 > root(2). The weakest champion (2) is dethroned. New champions: {3, 5}.

Pop 3

Push 6 → heap = [3, 5, 6] → pop 3 → heap = [5, 6]

6 > root(3). The weakest champion (3) is dethroned. New champions: {5, 6}.

Pop 4

Push 4 → heap = [4, 6, 5] → pop 4 → heap = [5, 6]

4 < root(5). The challenger (4) cannot beat the current weakest champion. It self-evicts. Champions unchanged: {5, 6}.

Answer

return heap[0] = 5 ✓

heap = [5, 6]. The root (minimum of the top-2) is 5. The 2nd largest in the array is 5.

06 · The Quickselect approach — O(n) average time

Finding the answer without examining the full sorted order

Quickselect is a variant of Quicksort. Instead of fully sorting the array, it partitions around a pivot and recurses into only the one half that must contain the target element. On average, each level halves the problem — giving O(n) total work.

import random
 
def findKthLargest(nums, k):
target = len(nums) - k # kth largest = (n-k)th smallest (0-indexed)
 
def quickselect(lo, hi):
pivot = nums[hi] # choose last element as pivot
p = lo # p = next slot for elements ≤ pivot
for i in range(lo, hi):
if nums[i] <= pivot:
nums[i], nums[p] = nums[p], nums[i]
p += 1
nums[p], nums[hi] = nums[hi], nums[p] # place pivot
 
if p == target: return nums[p]
elif p < target: return quickselect(p+1, hi)
else: return quickselect(lo, p-1)
 
random.shuffle(nums) # randomise to avoid O(n²) worst case
return quickselect(0, len(nums)-1)

Why target = len(nums) - k? Quickselect finds the (0-indexed) position in ascending sorted order. The k-th largest is the element at position n−k in ascending order. For example k=2 in n=6 → target index 4, which is 5 in [1,2,3,4,5,6].

Watching the partition recurse toward the answer

target = n−k = 6−2 = 4. We're looking for the element that would sit at index 4 in the sorted ascending array.

Pivot = 4

partition([3,2,1,5,6,4], lo=0, hi=5)

Pivot = nums[5] = 4. Scan: 3≤4 → left, 2≤4 → left, 1≤4 → left, 5>4 → right, 6>4 → right. Place pivot. Array: [3,2,1,4,5,6]. Pivot lands at index 3.

Compare

pivot_idx = 3, target = 4 → target is right of pivot

3 < 4 → recurse on the right half: quickselect(lo=4, hi=5). The left half [3,2,1,4] is discarded — the answer is not there.

Pivot = 6

partition([5,6], lo=4, hi=5)

Pivot = nums[5] = 6. Scan: 5≤6 → left. Place pivot. Sub-array: [5,6]. Pivot lands at index 5.

Compare

pivot_idx = 5, target = 4 → target is left of pivot

5 > 4 → recurse on left half: quickselect(lo=4, hi=4).

Found ✓

lo == hi == target == 4 → return nums[4] = 5

Single-element range — this element is the answer. Total work: scanned 6 + 2 = 8 elements instead of sorting all 6. Answer = 5.

08 · Real-world analogy

Think of a tournament leaderboard

Imagine you're running a leaderboard for a competition and you only care about the top-k scores. You have a physical board with k slots. When a new score comes in, you slot it in. If the board overflows to k+1 entries, you cut the weakest score. At the end, the weakest score still on the board is your k-th best. That's exactly what the min-heap does — the root is always the weakest champion currently holding a leaderboard slot.

09 · Common mistakes

Errors to avoid in interviews

✕

Using a max-heap instead of a min-heap

A max-heap would require storing all n elements then popping k times — O(n + k log n) with O(n) space. The min-heap of size k keeps only O(k) space and gives the answer at the root immediately. The type is counterintuitive: to find the k-th largest, use a min-heap.

✕

Confusing k-th largest with k-th distinct largest

In [5,5,5] with k=2 the answer is 5, not "there is no 2nd distinct value." Duplicates fill separate rank positions. The min-heap handles this automatically — push all values including duplicates.

✕

Off-by-one in Quickselect: target = k-1 instead of n-k

The k-th largest in descending order is the (n−k)-th element in ascending order (0-indexed). Writing target = k−1 finds the k-th smallest, not largest. Verifying with a small example before submitting catches this error instantly.

✕

Not randomising the pivot in Quickselect

Always choosing the last (or first) element as pivot degrades to O(n²) on sorted or reverse-sorted inputs. Adding random.shuffle(nums) before calling quickselect — or randomly selecting the pivot inside the partition — guarantees O(n) expected time on any input.

11 · Interview Q&A

Questions you'll actually get asked

When would you prefer Quickselect over the heap?
Quickselect is better when you can modify the input array and want O(1) extra space. It's also theoretically faster (O(n) vs O(n log k)) when k is large relative to n — say k = n/2 where the heap's log k term is log(n/2) ≈ log n. The heap is preferred for streaming data (elements arrive one-by-one, no random access) and when the input must not be mutated.
What is the worst case for Quickselect and how do you prevent it?
The worst case is O(n²) and occurs when the pivot is always the minimum or maximum of the remaining range — for instance, always picking the last element on an already-sorted input. The fix is to either shuffle the array randomly before starting (random.shuffle(nums)) or use "median of three" pivot selection. Both reduce the probability of worst-case behaviour to negligible.
How do you solve this if the input arrives as a stream (elements one at a time)?
Use the min-heap approach — it processes elements one by one and always maintains an up-to-date answer at the root. Quickselect requires the full array in memory. This is exactly what LC 703 (Kth Largest Element in a Stream) tests: you initialise the heap with the first batch and then call an add(val) method that does one push and optionally one pop.
What if k = 1 or k = n?
Both edge cases are handled correctly by all three approaches with no special-casing. k=1 means "find the maximum" — the heap keeps only 1 element and the root is the running max. k=n means "find the minimum" — the heap eventually holds all n elements and the root is the global minimum. The constraint guarantees 1 ≤ k ≤ n, so these are valid.

12 · Interview context

Where you'll see this pattern again

LC 703 — Kth Largest Element in a Stream

Direct extension of this problem to a data stream. Initialise the heap once, then for each new element call push + conditional pop. The answer is always the heap root.

LC 347 — Top K Frequent Elements

Same size-k min-heap pattern but keyed on frequency rather than value. Count frequencies first, then run the heap over (freq, element) pairs.

LC 973 — K Closest Points to Origin

Keep k closest using a max-heap (opposite direction from here): if new distance is smaller than the farthest in the heap, replace it. The heap root at the end holds the boundary distance.

LC 295 — Find Median from Data Stream

Uses two heaps together (max-heap for lower half, min-heap for upper half) to maintain a running median. The intuition is identical — each heap tracks its half and the boundary between them is the answer.
