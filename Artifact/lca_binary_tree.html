Lowest Common Ancestor of a Binary Tree

Find the deepest node that is an ancestor of both p and q — even when one might be the ancestor of the other.

Post-order DFS Tree Recursion Interview Essential

Key concepts at a glance — for those who already know the basics.

01 · The rule

First node from which both p and q are reachable (downward)

Run post-order DFS. Return a node only when you've confirmed targets on both sides — or when the node itself is p or q. The first node to receive non-null from both children is the LCA.

02 · How to think

1
Base case: if node is null, p, or q — return node
Null propagates "not found". Returning p or q itself signals "I found a target here".
2
Recurse left and right, collecting results
left = lca(node.left, p, q)  |  right = lca(node.right, p, q)
3
If both sides returned non-null → current node is the LCA
p is in one subtree, q is in the other. This node is the split point — return it.
4
Otherwise return whichever side is non-null
Both targets are in the same subtree — bubble the non-null result up.

03 · Code

def lowestCommonAncestor(root, p, q):
    # Base case: null → not found; p/q → signal "found target"
    if not root or root is p or root is q:
        return root
 
    left = lowestCommonAncestor(root.left, p, q)
    right = lowestCommonAncestor(root.right, p, q)
 
    # Both subtrees found a target → this node is the LCA
    if left and right:
        return root
 
    # Only one side found something → propagate it upward
    return left if left else right

06 · Complexity

Time
O(n)
Every node is visited exactly once during the post-order traversal. No extra passes.
Space
O(h)
Recursion stack depth equals tree height h. Worst case O(n) for a skewed tree; O(log n) for balanced.

01 · What does "lowest common ancestor" actually mean?

Defining the problem precisely

A common ancestor of two nodes p and q is any node that has both p and q as descendants (where a node is considered a descendant of itself). The lowest one is simply the deepest — the one farthest from the root that still qualifies.

The problem guarantees p and q both exist in the tree and are distinct. Notice that a node can be its own ancestor — so if p is a direct ancestor of q, then LCA(p, q) = p, not some higher node. This is the edge case that trips most people.

Visually: draw the path from root to p and the path from root to q. The LCA is the last node where these two paths share a common node before they diverge.

02 · Why post-order DFS?

The traversal order that makes this elegant

Post-order DFS means we process children before the parent. This is crucial: we need to know what both subtrees found before we can decide whether the current node is the LCA.

Post-order DFS ← natural fit

Recurse left, recurse right, then decide at the current node. Children report findings up to parent via return values. Clean and elegant.

Iterative with parent map

Store parent pointers for all nodes, collect ancestors of p in a set, then walk q's ancestors to find the first match. O(n) time but O(n) extra space and more code.

Interview tip: Always lead with the recursive approach. If the interviewer asks about iterative, mention the parent-pointer approach and note it requires O(n) extra space vs. O(h) for recursion stack.

03 · The real-world analogy

Two people searching for each other in a family tree

Imagine you're a detective hired to find the nearest shared ancestor of two people in a family tree. You send out two search parties — one to find person p, one to find person q. Each party reports back to their parent: "found p", "found q", or "found nothing". The first family member who hears back from both sides is the common ancestor you're looking for.

Key insight: The function doesn't just return the answer — it also acts as a messenger. When it finds p or q, it returns that node upward as a signal. A node receiving two non-null signals becomes the LCA and returns itself — which then propagates all the way up to the root unchanged, because no ancestor above will receive two non-null signals again.

04 · Understanding the code line by line

What every line is actually doing

The elegance of this solution is that it uses the return value for two different purposes simultaneously: as a search signal going up, and as the final answer once the LCA is locked in.

def lowestCommonAncestor(root, p, q):
    # 1. null → not in this branch → report "nothing found"
    # 2. root == p → signal "p is here" (don't go deeper — even if q is
    # below, p is still the LCA per problem definition)
    # 3. root == q → same logic, signal "q is here"
    if not root or root is p or root is q:
        return root
 
    # Post-order: get answers from both subtrees first
    left = lowestCommonAncestor(root.left, p, q)
    right = lowestCommonAncestor(root.right, p, q)
 
    # Both non-null → one target in each subtree → this is the split point
    if left and right:
        return root    # THIS node is the LCA
 
    # Only one side has something — bubble it up unchanged
    return left if left else right
Why do we return root immediately when root is p or q, without searching deeper?
Because the problem guarantees both p and q exist in the tree. If we've reached p, then q must be somewhere — either in p's subtree (making p the LCA) or in another part of the tree entirely. Either way, the correct answer is p itself or something above p. We don't need to know where q is; returning p as a signal is sufficient for the parent node to decide.
After the LCA is found and returned, what happens to all the calls above it?
Once the LCA (say node 3) is returned by lca(3, p, q), every ancestor of node 3 will receive that node as either its left or right result — but never both (since both targets are now "accounted for" in node 3's subtree). So the other side will return null, and the LCA node just gets propagated upward unchanged until it reaches the root call.
What if neither target is found in a subtree?
Both left and right will be null. The last line return left if left else right will return null (right), which correctly signals "nothing found here" to the parent node.
Can the return value be ambiguous — signaling a "found target" vs "this is the LCA"?
Yes — intentionally. The function is dual-purpose: before the LCA is found, returned nodes are signals ("target found here"). After the LCA is found, the returned node is the actual answer bubbling up. From the caller's perspective (root call), this doesn't matter — the final return is always the LCA.

06 · Common beginner mistakes

What trips people up

✕

Forgetting that a node can be its own ancestor

LCA(5, 4) in our example is 5 because 4 is in 5's subtree. The base case handles this correctly — we return 5 without searching its subtree, and 5 is propagated as the answer. Never special-case this — the algorithm handles it automatically.

✕

Comparing node values instead of node references

Use root is p (identity check), not root.val == p.val. In a tree, duplicate values can exist. The problem gives you actual node objects as p and q, so reference comparison is both correct and safe.

✕

Searching deeper after finding p (or q) to locate the other target

Unnecessary. Once you hit p, returning p is the correct signal. If q is a descendant of p, the parent of p will receive p from one side and null from the other, and will propagate p upward — which is exactly the correct LCA result.

✕

Using pre-order instead of post-order (checking node before recursing)

Pre-order returns as soon as it finds p or q — it can't know whether both are in one subtree or split across two. Post-order (recurse first, then decide) is essential here. The decision at a node depends on its children's results.

07 · Complexity — the full explanation

Why O(n) time and O(h) space?

Time
O(n)
Every node is visited exactly once. There's no revisiting or extra passes. In the best case (p or q is root), we return immediately — but worst case is still O(n).
Space
O(h)
The implicit recursion stack can be at most h frames deep, where h is tree height. O(log n) balanced, O(n) worst case (skewed tree).

Interview tip: When asked about space, distinguish between the recursion stack (O(h)) and the iterative parent-map approach (O(n) for the hash map). State clearly: "This recursive approach uses O(h) call stack space — O(log n) for a balanced tree, O(n) worst case." Interviewers appreciate the nuance.

08 · Interview context

Related problems to know

LCA of BST (LeetCode 235)

With a BST, you don't need full post-order DFS. Use node values: if both p and q are less than root, go left; both greater, go right; split means current node is LCA. O(h) time and O(1) space (iterative).

LCA with parent pointers

Walk from p to root collecting ancestors in a set. Then walk from q to root and return the first node already in the set. O(h) time and O(h) space. Useful when the tree gives you parent pointers.

LCA of Deepest Leaves (LeetCode 1123)

Same post-order DFS structure, but instead of targets p and q, you track depth and return the LCA of all nodes at max depth. A natural extension of this technique.

Distance Between Two Nodes

dist(p, q) = depth(p) + depth(q) − 2 × depth(LCA(p, q)). Find the LCA first using this algorithm, then compute depths. Classic multi-step tree problem.
